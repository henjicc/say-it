# 实施方案

## 当前项目和相关功能现状

「说吧！」是 Windows 优先的桌面端语音输入与实时字幕应用，技术栈为 Tauri 2 + Rust 2021 后端、React 19 + TypeScript + Vite 6 + Tailwind CSS 4 前端，包管理器为 npm。主窗口入口为 `ui/index.html`，悬浮指示/字幕窗口入口为 `ui/indicator.html`。

当前前端核心目录：

- `ui/src/App.tsx`：主窗口应用壳层，负责主题变量注入、标题栏、侧栏、主内容视图切换。
- `ui/src/index.css`：Tailwind 4 入口与当前基础设计变量。
- `ui/src/components/shell/`：标题栏、侧栏、导航配置。
- `ui/src/components/ui/`：Button、Card、Tabs、Input、Select、Slider、Collapse、Field 等基础组件。
- `ui/src/views/`：语音输入、实时字幕、录音识别、文稿对齐、设置页及子面板。
- `ui/src/features/`：听写、字幕、录音识别、音频设备、文本处理等业务控制逻辑。
- `ui/src/store/`：Zustand 状态与偏好设置。
- `ui/src/indicator/`：悬浮指示窗口与实时字幕悬浮窗样式。

当前参考资源：

- 参考设计图：用户提供的语音输入、实时字幕、录音识别三张目标视觉图。
- 参考代码：`docs/others/stitch_speak_bar_ui_reconstruction/_1/code.html`、`_2/code.html`、`_3/code.html`。
- 当前截图：`docs/images/语音输入.png`、`docs/images/实时字幕.png`、`docs/images/设置.png`。

## 当前实现方式

主窗口使用无边框窗口和自定义标题栏。`App.tsx` 通过 `useThemeStore` 写入 CSS 变量，页面通过 `useUiStore` 在 `dictation`、`subtitles`、`transcription`、`settings` 之间切换。多数页面已使用项目内 UI 组件，但仍有较多业务组件直接写 Tailwind 颜色、间距、圆角和透明度。

Select、Tabs、Button、Slider 等基础组件已经存在，可继续复用和扩展。录音识别和文稿对齐依赖文件选择、拖放、Tauri dialog、后端命令和导出逻辑；实时字幕依赖字幕偏好、音频设备、快捷键捕获、悬浮窗同步；语音输入依赖快捷键捕获、麦克风设备、注入方式和本地文本处理。

## 当前存在的问题

- Design Token 不完整，业务组件中散落 `white/10`、`bg-white/[0.03]`、`rounded-xl`、固定网格和状态色。
- 当前 UI 比参考图更偏纯黑和空旷，桌面端信息密度、对齐节奏、分组层级不足。
- 页面缺少统一的 `PageHeader`、`SectionHeader`、`SettingsSection`、`StatusBar` 等布局组件。
- 基础组件视觉规格不够统一，按钮圆角、表单高度、禁用态、hover、focus、active、浮层阴影仍需规范。
- 设置页、音频调试、本地规则等高密度界面与主页面视觉体系不完全一致。
- 鼠标和桌面端交互需要保留，但部分交互只能由用户人工验证。

## 最终目标

在不破坏现有功能的前提下，建立统一、专业、适合桌面端软件的信息架构和视觉系统。改造后的界面应接近参考图的方向：深色背景、克制蓝色强调、固定左侧导航、稳定页头、分段页签、带蓝色竖线的分组标题、两列高密度表单、统一控件状态和更清晰的状态反馈。

## 总体解决思路

1. 先统一底层设计变量和基础组件，不直接重写页面。
2. 再将核心页面逐步迁移到统一页面骨架和分组布局。
3. 最后处理设置、调试、本地规则等复杂界面，并完成尺寸、构建和人工交互验证。

## 技术方案或实施策略

- 使用 `ui/src/index.css` 集中维护颜色、文字、边框、背景、圆角、阴影、间距、动效等 CSS Variables。
- 保留 Tailwind CSS 4 写法，但页面组件优先引用 token 或统一组件 class，减少硬编码视觉值。
- 扩展现有 `ui/src/components/ui`，必要时新增布局组件，不引入大型 UI 框架。
- 保留当前 store、feature controller、Tauri 命令和事件，不因 UI 改造调整数据流。
- 对参考 HTML 只提取布局关系和控件密度，不复制 CDN、外部图标、静态文件输入或不可用链接。

## 任务范围

- 主窗口整体壳层、标题栏、侧栏、主内容容器。
- Design Token 与基础 UI 组件。
- 语音输入、实时字幕、录音识别、文稿对齐页面。
- 设置页下的密钥与识别、录音调整、启动设置、麦克风与提示音、外观。
- 音频调试、本地处理规则、热词、高级参数、折叠面板等复杂控件。
- 构建验证、尺寸验证、人工交互验证步骤整理。

## 非任务范围

- 不重写 Rust 后端、ASR 服务、音频采集、系统托盘、窗口控制业务逻辑。
- 不改变 Tauri invoke/event 名称和现有数据结构。
- 不新增大型 UI 框架或替换 Tailwind。
- 不处理 `docs/task/录音识别与文稿对齐`。
- 不擅自新增参考图中的“帮助与反馈 / 关于说吧！”真实入口；是否新增待确认。
- 不发布、不部署、不推送远端。

## 阶段划分

第一阶段先建立设计系统与应用壳层，是后续所有页面改造的基础。第二阶段改造用户最常用的核心页面。第三阶段处理设置页和复杂控件，降低视觉割裂。第四阶段集中做构建、尺寸、人工交互验证和文档收尾。

## 阶段之间的关系

- 1.1 是所有 UI 改造的基础。
- 1.2 依赖 1.1，并为 2.x、3.x 提供统一组件和应用骨架。
- 2.1、2.2、3.1、3.2 均依赖 1.2，彼此可以按模块并行。
- 4.1 依赖主要页面改造完成。
- 4.2 依赖 4.1，并负责人工交互验证清单和计划收尾。

## 新旧实现的兼容方式

改造期间保留现有组件导出名称和主要 props，优先在组件内部升级样式，减少业务页面改动范围。必须改动页面结构时，只移动展示层结构，不改变 store 字段、controller 调用、事件监听和异步流程。

## 数据或接口兼容要求

- `useUiStore` 的 view key 不变。
- `useThemeStore` 的主题持久化能力保留。
- `useDictPrefs`、`useDictationStore`、`useSubtitleStore`、`useTranscriptionStore`、`useProviderStore` 的数据结构不因 UI 改造改变。
- `CMD` 中的 Tauri 命令调用不改名、不改变参数形状。
- 文件拖放、文件选择、字幕预览、快捷键捕获等交互的调用路径不改变。

## 风险控制方式

- 每个任务只处理对应页面或组件范围，不顺带重构业务逻辑。
- 先做低风险 token 和组件，再做页面迁移。
- 每个任务完成后运行与改动范围匹配的检查，至少执行 `npx tsc --noEmit` 或 `npm run ui:build` 中的相关项。
- UI 交互改动需要提供人工验证步骤，不能仅凭编译通过判断完成。
- 修改前后检查 `git diff`，确保没有混入无关文件。

## 回退或恢复原则

每个任务应形成独立提交，便于按任务回退。若某个页面改造造成业务流程异常，应优先回退该页面改动，保留已稳定的 token 和基础组件改造。不得通过删除功能、吞掉异常或关闭类型检查来掩盖问题。

## 全局开发约束

- 默认中文文档和提交信息。
- 除颜色选择器、用户自定义颜色等特殊场景外，业务组件不直接写具体十六进制颜色。
- 不把复杂业务流程放进 React 展示组件；业务能力继续通过 `features`、`store`、`lib/tauri.ts` 使用。
- 多入口页面需同时考虑 `ui/index.html` 主窗口和 `ui/indicator.html` 悬浮窗。
- 不手工维护 `ui/dist/`、`src-tauri/target/`、`src-tauri/gen/schemas/`。

## 最终验收条件

- 主窗口主要页面视觉方向与参考图一致，且保留现有功能入口。
- 基础组件和页面布局使用统一 token，明显减少散落硬编码视觉值。
- 1366×768、1440×900、1920×1080 下主页面无明显重叠、截断、横向溢出。
- `npx tsc --noEmit` 和 `npm run ui:build` 通过，或明确记录无法执行的原因。
- 用户完成鼠标交互验证清单后，没有发现阻断级交互问题。
- 所有任务文件、总览和重要记录已同步更新。
