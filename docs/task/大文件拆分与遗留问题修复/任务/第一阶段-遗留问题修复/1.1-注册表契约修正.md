# 1.1 注册表契约修正（subtitles 场景消费与 providerKind 改名）

## 任务状态

- 状态：待验证（代码与自动化验证已完成，等待用户实测字幕面板下拉）
- 所属阶段：第一阶段-遗留问题修复
- 前置任务：无
- 关联任务：1.2（同阶段，文件不重叠）
- 是否可以并行：可与 1.2 并行

## 任务目标

让模型注册表 `shared/asr-models.json` 的两个失真契约恢复一致：

1. `scenes` 中的 `subtitles` 场景有真实消费方——字幕面板的模型下拉从 `subtitles` 场景派生，而不是复用听写实时场景。
2. `providerKind` 字段改名为 `providerId`，名实相符（它存的是 provider id `"funasr"`，不是 profile 的 kind `"alibabacloud-funasr"`），并同步更新操作手册。

## 当前情况

- 前端模型选项统一从注册表派生：`ui/src/features/asr/modelRegistry.ts`（`optionsForScene` 等查询函数）→ `ui/src/features/asr/modelOptions.ts`（导出各场景常量）→ 各视图消费。
- 字幕面板 `ui/src/views/RealtimeSubtitlesPanel.tsx:165` 用的是 `REALTIME_ASR_MODEL_OPTIONS`（= `optionsForScene("dictationRealtime")`），全仓库没有 `optionsForScene("subtitles")` 调用。当前所有实时模型两个场景都声明了，所以下拉内容暂时一致，但按 `docs/rules/新增供应商与模型操作手册.md` 场景一新增「只标 subtitles」的模型时，行为将与手册承诺不符。
- `providerKind` 字段在后端 `src-tauri/src/providers/registry.rs` 的 `ModelInfo.provider_kind` 与前端 `modelRegistry.ts` 的 `ModelInfo.providerKind` 中解析，但都没有逻辑消费方；手册场景一步骤 1 与场景三步骤 3 提及该字段。
- 后端 `registry.rs` 的 `ModelInfo` 整体 `#[allow(dead_code)]`，改字段名只需同步 serde 的 camelCase 映射（`provider_id` → `providerId`）。

## 实施方案

纯数据契约修正，不改任何识别行为：

- 在 `modelOptions.ts` 新增 `SUBTITLE_ASR_MODEL_OPTIONS = optionsForScene("subtitles")`，字幕面板改用它。当前数据下下拉内容与改前完全一致（两场景模型集合相同），属于「派生来源修正」而非行为变化。
- `shared/asr-models.json` 全部记录的 `providerKind` 键改为 `providerId`；后端 `ModelInfo.provider_kind` 改为 `provider_id`；前端 `ModelInfo.providerKind` 改为 `providerId`；手册同步改写字段说明。

### 补充修复：全局快捷键行的"清除"按钮挤占空间（用户实测中发现，历史遗留问题，非本任务原定范围，顺带修复）

- 背景：用户实测实时字幕面板时发现，设置快捷键后该行会同时出现「修改/取消」与「清除」两个按钮，快捷键较长（如 `Ctrl + Alt + Shift + F12`）时输入框被挤压导致文字显示不全。
- 方案确认：与用户讨论后否决了"单按钮复用、捕获中未按键点取消即清空"的方案（会导致点击"取消"与按 Esc 取消捕获两种操作语义不一致，容易误清空），采用用户认可的推荐方案——清除功能挪进输入框内嵌的"×"图标，主按钮固定只做"修改/取消"一件事，语义不变。
- 修复：`ui/src/views/RealtimeSubtitlesPanel.tsx` 移除独立的「清除」按钮；输入框外层加 `relative` 包裹并在 `!capturing && shortcutLabel` 时显示内嵌的 `ClearIcon` 按钮（右侧，点击调用原有 `clearSubtitleShortcut`），输入框始终只搭配一个「修改/取消」按钮，不再随快捷键设置状态增减按钮数量。
- 验证：`npx tsc --noEmit`、`npm run ui:build` 均通过。
