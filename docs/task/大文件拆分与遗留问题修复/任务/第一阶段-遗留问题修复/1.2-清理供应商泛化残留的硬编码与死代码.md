## 任务目标

清除《供应商与模型扩展性改造》泛化后残留的三处单供应商假设与一处死代码，使涉及文件在被第二、三阶段拆分前处于干净状态：

1. `commands/asr.rs` 静音测试的重复 `TaskFinished` match 分支（死代码）。
2. `commands/asr.rs` 静音测试错误前缀写死 `"Fun-ASR 上游错误"`。
3. `ui/src/features/dictation/controller.ts` 文件听写报错写死 `"请先在设置中保存阿里云百炼 API Key"`。
4. `ui/src/views/SettingsProviderPanel.tsx` 取「第一个 enabled 的 asr profile」而非生效供应商。

## 当前情况

- `src-tauri/src/commands/asr.rs:494-495`：`AsrEvent::TaskFinished if started => break,` 与 `AsrEvent::TaskFinished => break,` 两分支行为完全相同，guard 无效，是连接器 trait 化重构（提交 1c3018b）的遗留。
- `src-tauri/src/commands/asr.rs:497`：`run_realtime_silence_test` 的 `TaskFailed` 分支返回 `format!("Fun-ASR 上游错误 [{code}]: {message}")`；该函数已通过 `realtime_connector_for` 泛化到任意供应商，前缀不该写死。函数签名里有 `profile: &ProviderProfile`，可取 `profile.display_name`。
- `ui/src/features/dictation/controller.ts:229`（`startFileDictation` 内）：`throw new Error("请先在设置中保存阿里云百炼 API Key")`；而它调用的 `ensureDictationProviderReady()`（同文件约 91 行）已改为按 `effective("asr")` 判断，文案与判断逻辑脱节。`useProviderStore` 提供 `labelFor(providerId)` 可取供应商显示名。
- `ui/src/views/SettingsProviderPanel.tsx:58`：`providers.find((p) => p.enabled && p.capabilities.includes("asr"))` 取第一个 asr profile。后端 `normalize_settings`（`src-tauri/src/providers/mod.rs`）现在保留未知 profile 且内置 profile 缺失时是 push 到末尾，极端情况下（用户手工在 `say-it-state.json` 加了带 asr 能力的 profile 且排在 funasr 前）面板会渲染错 profile，并把 API Key 经 `updateProviderConfig(provider.id, ...)` 写进错误对象。
- 同文件还有一个命名遗留可顺带处理：`commands/asr.rs` 的 `funasr_event_to_value` 函数实际处理的是协议无关的 `AsrEvent`，名字仍带 funasr。

## 实施方案

逐点最小修复，不引入新抽象：

- 死代码：合并为单个 `AsrEvent::TaskFinished => break,` 分支；`started` 变量若因此再无读取方则一并移除（执行时以 `cargo check` 警告为准）。
- 静音测试文案：前缀改为 `format!("{} 上游错误 [{code}]: {message}", profile.display_name)`。
- 听写文案：`startFileDictation` 的错误消息改为基于 `useProviderStore.getState().labelFor(effective("asr"))` 生成，如 `` `请先在设置中保存 ${label} API Key` ``；`labelFor` 对未知 id 回退 id 本身，无供应商时用通用文案「请先在设置中配置识别供应商 API Key」（执行时确认 `effective` 为空串的分支）。
- 设置面板：改为先按 `defaults.asr`（store 的 `effective("asr")`）定位 profile，找不到再回退第一个 enabled asr profile；行内注释同步更新。视觉不变。
