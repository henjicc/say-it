# 供应商与模型扩展性改造 · 重要记录

## 记录 001：模型注册表的载体方案

- 日期：2026-07-06
- 状态：待确认
- 背景：模型能力信息需要前后端共享单一数据源。Rust 与 TS 无共享类型系统，两种可行方案：
  - **方案 A（推荐）**：仓库根 `shared/asr-models.json`，Rust 用 `include_str!` 编译期内嵌 + 启动时 serde 解析，前端直接 import（Vite 内联进 bundle）。优点：真正单一数据源、无运行时加载时序问题、两侧都是同步查询；缺点：两侧各需一层薄封装与字段约定（靠 1.1/1.2 的单测互相锁定）。
  - **方案 B**：注册表只存在于 Rust，新增 Tauri 命令 `list_asr_models` 下发给前端，前端启动时加载进 store。优点：前端零数据文件；缺点：现有 `modelOptions.ts` 的消费方全是同步调用，改成异步加载会波及 10 个文件的使用方式，改造成本明显更高。
- 最终决定：（待用户确认，默认按方案 A 执行）
- 选择原因：方案 A 对现有同步消费方式零侵入，改动面最小。
- 影响任务：1.1、1.2、4.1

## 记录 002：文件转写协议不在本次抽象范围

- 状态：已确定

- 最终决定：本次只把"模型→通路"的判断数据挪进注册表（任务 1.1），通路实现继续留在 `providers/alibabacloud/transcription.rs` 内按分支组织；将来真要接入第二家的文件转写时再抽 trait。
- 选择原因：现在抽象只有一个实现方，属于过早抽象；注册表已保证"加模型"场景低成本，"加供应商的文件转写"是低频事件。
- 影响任务：1.1、4.1（手册需写明此边界）

## 记录 003：LLM 后处理只预留机制，不实现功能

- 背景：用户计划后续引入大语言模型对识别结果做更细致的优化，希望本次改造顺带考虑。
- 最终决定：本次仅做能力机制预留——`ProviderDefaults` 增加 `llm` 字段（3.1）、前端 `ProviderCapability` 类型含 `"llm"`（3.2）、`has_capability` 等函数数据驱动不再写死 "asr"。不新增任何 LLM 供应商 profile、不做后处理管线、不做 UI。将来接入时：新建 provider 模块（如 `providers/<llm供应商>/`）+ 内置 profile 声明 `capabilities: ["llm"]` + 新命令层，即可挂进现有机制；届时另立任务计划。
