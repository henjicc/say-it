# 3.1 解除后端单供应商锁与默认值多能力化

## 任务状态

- 状态：已完成
- 所属阶段：第三阶段-多供应商基础设施
- 前置任务：2.1
- 关联任务：3.2（前端同步泛化）
- 是否可以并行：否（动持久化结构，需独占）

## 任务目标

后端不再假设"只有 funasr 一个供应商"：`normalize_settings` 保留而非删除未知 profile；命令层按 profile 的 `kind` 分发而非硬编码；`ProviderDefaults` 结构可表达多能力默认值（为 llm 预留）。旧 `say-it-state.json` 无损兼容。

## 当前情况

- `src-tauri/src/providers/mod.rs:115-136` `normalize_settings`：
  - `retain(|p| p.id == FUNASR_PROVIDER_ID)` 删除一切非 funasr profile；
  - 强制覆写 `display_name = "阿里云百炼"`、`enabled = true`、`capabilities = ["asr"]`；
  - 该函数在读、写、每次设置变更时都会执行（`persistence.rs:61,85`、`commands/common.rs:35`）。
- `src-tauri/src/providers/mod.rs:138-153`：`has_capability` / `default_provider_id` 写死 `capability == "asr"` 与 funasr 兜底；`set_default_provider` 拒绝非 asr 能力。
- `commands/asr.rs:30`：`profile.kind != "alibabacloud-funasr"` 直接报错（2.1 完成后此处应已是"工厂建连接器"，本任务把工厂选择改为按 `kind` 查供应商模块）。
- `commands/transcription.rs:304-315`（`funasr_api_key`）、`320-337`（`funasr_hotword_state`）：绕过 `resolve_provider_id` 直接 `find_profile(FUNASR_PROVIDER_ID)`。文件转写目前只有阿里云实现，改造后仍可只支持阿里云，但取 profile 必须走"默认/指定供应商解析 + kind 校验"，不支持的 kind 给明确报错。
- `commands/funasr.rs`：热词命令是阿里云专属功能（对应设置页 `FunAsrHotwordsPanel.tsx`），保留按 ID 定位，但注释写明这是"供应商专属命令"的既定模式。
- 持久化兼容基线（旧 JSON 形态）：`{"providers":{"profiles":[{"id":"funasr","kind":"alibabacloud-funasr",...,"config":{"apiKey":"...","hotwords":[...],"vocabularyIds":{...}}}],"defaults":{"asr":"funasr"}}}`。
- `ProviderSettings` 的 `Default` impl 与 `funasr_profile()` 在 `providers/mod.rs:34-43,89-109`。

## 实施方案

1. **内置供应商注册**：`providers/mod.rs` 新增 `builtin_profiles() -> Vec<ProviderProfile>`（现在只有 `funasr_profile()`）。`normalize_settings` 改为：
   - 对每个内置 profile：不存在则插入；存在则仅修正**固定字段**（`kind` / `display_name` / `capabilities`），`config` 与 `enabled` 保留用户值（`enabled` 现状强制 true，维持现状并注明原因：UI 尚无停用开关）；
   - 未知 id 的 profile **保留原样**（为未来供应商与用户手工配置留空间）；
   - defaults 校验逻辑不变：默认供应商无效时回退到第一个具备该能力的内置 profile。
2. **defaults 多能力化**：`ProviderDefaults` 改为 `{ asr: String, #[serde(default)] llm: String }`（llm 空串表示未设置；不用 map，保持序列化格式直观且旧 JSON 天然兼容）。`has_capability` / `default_provider_id` / `set_default_provider` 改为按能力字符串通用处理（能力合法性以 profile 的 capabilities 数据为准，不再写死 "asr"）。
3. **命令层分发**：`start_asr_stream` 的连接器工厂改为 `providers::realtime_connector_for(kind, config, model)`（match kind → 供应商模块工厂，未知 kind 报"当前版本不支持供应商类型：{kind}"，即保留现有文案）；`commands/transcription.rs` 两个取配置函数改为 `resolve_provider_id(state,"asr",None)` + kind 校验后读取。
4. **兼容测试**：用上面旧 JSON 基线字符串写反序列化 + `normalize_settings` 单元测试，断言 apiKey / hotwords / vocabularyIds / defaults.asr 全部保留。
