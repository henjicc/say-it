# 业务后端化与托盘轻量化 · 实施方案

## 当前项目和功能现状

### 技术与运行结构

- 桌面壳：Tauri 2，Windows 为主要目标；主窗口与 `dictation-indicator` 悬浮窗都是 WebView2。
- 后端：`src-tauri/src` 已负责音频设备、CPAL 采集、ASR WebSocket、文件转写、翻译请求、OBS 连接、文本注入、持久化和桌面窗口能力。
- 前端：`ui/src` 约 102 个 TS/TSX/CSS 文件、约 1.28 万行。页面和组件约占一半，另外约 4800 行位于 `features/`，包含大量会话状态机和业务编排。
- 前后端边界：前端通过 `ui/src/lib/tauri.ts` 中约 60 个命令与 20 余个事件调用 Rust；`ui/src/hooks/useTauriBridge.ts` 是主窗口级事件总线。
- 持久化分裂：供应商、快捷键、启动项、字幕翻译模型和 OBS 配置在 `say-it-state.json`；听写偏好、字幕样式、模型对比、主题和自定义提示音在 WebView `localStorage`。
- 既有扩展机制：`shared/asr-models.json` 是模型注册表；`providers/connector.rs` 已有实时识别连接器 trait；`providers/alibabacloud/` 是当前唯一供应商实现。

### 当前前端承担的非展示职责

| 领域 | 当前主要位置 | 当前职责 | 目标归属 |
|---|---|---|---|
| 听写 | `features/dictation/` | 快捷键回退、音频会话、静音门控、实时/文件识别、收尾、规则处理、提示音、注入 | Rust 应用服务 |
| 实时字幕 | `features/subtitles/controller.ts` | 音源选择、ASR 重连、断流重连、分句翻译、显示文本合并、OBS 路由、预览编排 | Rust 应用服务；动画渲染留前端 |
| 转写与对齐 | `features/transcription/`、`store/useTranscriptionStore.ts` | 任务监听、阶段推进、识别缓存、字幕 cue 派生、导出 | Rust 服务与文档模型；编辑手势留前端 |
| 模型对比 | `features/compare/controller.ts` | 多模型会话、音频缓冲、并发状态、播放进度 | Rust 应用服务 |
| 音频调试 | `features/audio/lab.ts` | 录音、PCM 缓冲、重处理调度、播放 | Rust 会话；Canvas 绘制留前端 |
| 配置 | 多个 Zustand store | 默认值、归一化、兼容迁移、持久化 | Rust；前端只缓存快照和表单草稿 |
| 模型与供应商 | `features/asr/`、`useProviderStore.ts` | 前端直接读取注册表并判断能力 | Rust 目录服务 |

## 当前存在的问题

1. **主窗口不能销毁**：全局快捷键只是由 Rust 发事件，真正的听写/字幕启动逻辑在 `useTauriBridge` 和前端 controller 中。销毁主 WebView 后事件无人消费。
2. **运行时状态分散且不可恢复**：大量模块级变量保存在 TypeScript 内存中，重新创建 WebView 后无法还原当前会话。
3. **业务逻辑重复跨层**：模型类别、配置归一化、音频 RMS、字幕切分和状态文案等同时存在于 Rust 与 TypeScript。
4. **新增供应商仍会碰前端**：实时连接器已有抽象，但文件识别、翻译、热词命令和供应商设置 UI 仍带阿里云/`funasr` 专属入口。
5. **数据来源不统一**：`say-it-state.json` 与四类 `localStorage` 键分别维护，无法由无界面后台进程完整恢复。
6. **WebView2 多进程是结构性成本**：仅调用 `hide()` 不会释放主 renderer、GPU 和公共服务进程；要显著降低托盘内存，必须允许销毁主 WebView。

## 最终目标

### 功能目标

- 没有主 WebView 时，全局听写、按住说话、取消、实时字幕、翻译、OBS 输出和后台转写仍由 Rust 正常运行。
- 重新打开主窗口时，通过后端状态快照立即恢复页面状态，不依赖旧 WebView 内存。
- 主窗口关闭到托盘后可以销毁并按需重建；窗口位置、尺寸、主题、配置和运行状态保持一致。
- 新增同协议模型仍以注册表为主；新增供应商时，主要工作集中在 `providers/<供应商>/`、能力实现和配置描述，不再修改业务 controller。
- 前端页面只负责展示、收集用户意图和高频视觉交互，不持有后台业务状态机。

### 架构目标

建议形成以下依赖方向，具体文件名可在任务执行时根据代码规模微调：

```text
React 视图与组件
    ↓ 命令 / 状态快照 / 领域事件
commands/（薄 Tauri 传输层）
    ↓
application/（听写、字幕、转写、对比、音频调试、设置、生命周期）
    ↓
providers/（供应商适配）  desktop/（Windows/音频/窗口/注入）  persistence.rs
```

- `commands/` 只解析参数、调用应用服务、转换错误，不保存跨调用业务状态。
- `application/` 持有状态机、取消控制、重连策略和领域事件，是运行时唯一权威。
- `providers/` 只处理供应商协议与能力，不依赖 React、窗口或业务页面。
- `desktop/` 封装平台和设备能力，不反向依赖应用服务。
- 前端 Zustand store 降为后端快照的只读缓存和短生命周期 UI 状态。

## 总体解决思路

### 1. 先建立契约，再迁移领域

先固化当前行为、内存和时延基线，定义统一状态快照和事件信封。事件至少包含领域、事件类型、会话 ID/代次和递增版本，避免窗口重建或异步迟到事件覆盖新状态。前后端统一使用 `camelCase` 序列化，Rust 类型是契约源。

### 2. 先迁移持久化，保证窗口可重建

扩展 `PersistedData`，加入 schema 版本以及听写偏好、字幕偏好、模型对比偏好、主题和提示音元数据。首次升级由仍存在的前端读取旧 `localStorage`，调用一次性导入命令；后端归一化并原子保存。迁移期保留旧键并在设置修改时维持兼容镜像，直到 5.1 明确结束兼容窗口。

### 3. 供应商能力先于业务运行时收敛

- `shared/asr-models.json` 可继续作为仓库数据源，但只由 Rust 解析并通过目录 API 输出给前端，前端不再直接导入和自行判断协议。
- 保留现有 `RealtimeAsrConnector`，补充文件识别、文本翻译、热词/词表等能力接口或等价的供应商服务边界。
- 配置字段由后端提供稳定描述；普通字段由通用表单渲染，复杂供应商能力允许专用扩展面板，避免为了“完全动态表单”过度设计。

### 4. 按领域切换唯一权威

每个领域采用“Rust 新路径完成并验证 → 前端切换消费快照/事件 → 删除旧 controller”的顺序。迁移过程中不得让新旧状态机同时订阅同一快捷键、音频流或 ASR 事件。必要时使用明确的内部所有权开关，但开关只服务于迁移和回退，最终必须删除。

### 5. 最后启用主窗口销毁

只有听写和字幕已不依赖主 WebView、所有必要配置可从 Rust 恢复后，才把关闭动作从 `hide()` 改为“保存窗口状态并销毁主窗口”。托盘打开时从 Tauri 配置重建窗口，前端启动后先拉取状态快照，再订阅增量事件。

悬浮指示窗先保留 WebView 实现以控制风险。主窗口销毁后的实测若已达到内存目标，则不继续改；若仍不达标，再评估“悬浮窗空闲销毁 + ready 握手 + 状态队列”。不得以牺牲首次快捷键响应或字幕稳定性换取任务管理器数字。

## 前端最终保留范围

以下内容保留在前端是有意设计，不视为迁移不彻底：

- 页面布局、组件组合、主题 CSS 变量应用、弹窗和导航状态。
- 输入框编辑中的草稿、焦点、展开/折叠、悬浮等纯 UI 状态。
- 字幕编辑器的拖拽、缩放、吸附预览、播放头动画等高频交互；领域修改在操作提交时同步到后端。
- Canvas/DOM 波形绘制和字幕文字动画；Rust 只提供经过压缩的视图数据，不通过 IPC 高频传输完整 PCM。
- 文件拖放和浏览器可直接完成的显示格式化；文件合法性和业务参数仍由后端再次校验。

## 任务范围

- 建立 Rust 应用服务层、状态快照和领域事件。
- 迁移前端持久化配置与兼容导入。
- 后端统一提供模型/供应商目录和能力调用。
- 迁移听写、字幕、转写/对齐、对比和音频调试编排。
- 实现主窗口按需销毁、重建和托盘性能测量。
- 删除被替代的旧 controller、事件和命令，更新相关规则文档。

## 非任务范围

- 改用 Slint、Qt、WinUI 等其他 GUI 框架。
- 重做当前视觉设计、导航结构或字幕编辑器交互。
- 接入新的真实模型、供应商或 LLM 后处理功能。
- 重写已经稳定的 ASR 协议解析、音频 DSP、文本对齐算法，除非迁移边界确实需要薄适配。
- 为追求更小数字调用 `EmptyWorkingSet`、禁用 GPU 或削弱 WebView2 安全隔离。

## 阶段划分与关系

### 第一阶段：基线与状态归一

1.1 固化现有行为、性能基线、生命周期语义和测试替身边界；1.2 将分裂的配置迁到 Rust 并提供完整快照。1.1 是所有任务的共同前置。

### 第二阶段：供应商能力后端化

2.1 把模型和供应商目录变成后端单一输出；2.2 在此基础上补齐文件识别、翻译、热词等能力边界。1.2 与 2.1 在 1.1 完成后可以并行。

### 第三阶段：后台实时核心迁移

3.1 先迁移共享音频会话和听写状态机；3.2 复用该基础迁移字幕、翻译和 OBS；3.3 在两个核心领域都脱离主 WebView 后实现真正托盘轻量模式。

### 第四阶段：其余业务流程迁移

4.1 迁移转写/对齐编排和可复用字幕派生；4.2 迁移模型对比和音频调试。4.1 可在 2.2 后与第三阶段部分并行；4.2 依赖 3.1 的共享音频会话。

### 第五阶段：前端收敛与总体验收

删除兼容期旧路径，检查前端只保留明确允许的职责，完成全功能回归、内存/CPU/唤起时延测量和新增供应商演练。

## 新旧实现兼容方式

- 每个领域独立迁移和提交，旧路径保留到新路径通过自动检查与用户实测后再删除。
- 同一领域不能双写运行状态；配置可以在兼容期双写 Rust 持久化与旧 `localStorage` 镜像。
- 新持久化字段全部 `#[serde(default)]`，未知字段保持可忽略；保存前保留上一份可恢复备份，写入采用临时文件替换。
- 前端启动顺序固定为：获取完整快照 → 建立事件监听 → 再次检查快照版本或执行增量补偿，避免订阅窗口内丢事件。
- 既有 Tauri 命令和事件在消费方迁移前保持兼容；最终删除项集中在 5.1，不在中途随手清理。

## 数据与接口兼容要求

- 必须无损读取现有 `say-it-state.json`，包括旧 identifier 目录兼容逻辑。
- 必须迁移并保留：`sayItDictPrefs`、`sayItSubtitlePrefs`、`sayItComparePrefs`、`sayItAccentTheme` 和自定义提示音数据。
- 模型 ID、供应商 ID、快捷键编码、字幕偏好字段和转写结果字段不得无迁移改名。
- 本地规则要保持规则顺序、启用状态、替换语义和错误提示；JavaScript 正则与 Rust 正则不完全兼容，具体引擎在 3.1 前确认。
- 主窗口重建后不得重复注册快捷键、重复启动麦克风、重复订阅 ASR 或重复发布 OBS 快照。

## 风险控制方式

| 风险 | 控制方式 |
|---|---|
| 迁移后听写/字幕时序改变 | 先为状态机补单元测试；会话使用代次与取消标记；逐领域切换唯一权威 |
| 配置丢失或回退后读取不到 | 版本化 schema、旧键保留、兼容镜像、原子写入、保存前备份 |
| async 锁和死锁 | 不跨 `.await` 持有 `MutexGuard`；按领域拆锁；事件发布在解锁后执行 |
| IPC 过多导致波形/字幕卡顿 | 后端聚合数据；高频视觉计算留前端；事件节流并带最新版本 |
| 供应商抽象过度设计 | 以现有阿里云通路和一个测试替身验证接口，只抽象真实差异；复杂配置允许扩展面板 |
| 主窗口销毁后功能退化 | 3.3 前不得启用销毁；所有快捷键和后台会话先通过无主窗口测试 |
| 悬浮窗销毁造成首次显示延迟 | 默认先保留指示窗 WebView；只有测量不达标时才试验空闲销毁，失败即放弃 |

## 回退与恢复原则

- 每个任务独立 Git 提交；任务失败优先 `git revert` 对应提交，不重写共享历史。
- 数据结构变更前先备份现有状态文件；迁移完成前不删除旧 `localStorage`。
- 领域切换必须保留明确的回退点。若新路径连续两次出现无法解释的行为偏差，停止迁移并恢复旧权威，不继续叠补丁。
- 3.3 的窗口销毁逻辑必须能单独回退为当前 `hide()`，不影响已经完成的业务后端化。

## 全局开发约束

- 遵守仓库 `AGENTS.md`：最小影响、解决根因、阶段性验证和每任务独立提交。
- 不新增依赖，除非现有标准库/依赖无法满足；新增前在当前任务记录体积、维护、安全和许可影响。
- Rust 模块职责需能一句话说明；应用层不得依赖 React/Tauri WebView 细节，Tauri 类型尽量止于命令和桌面适配层。
- 不执行全仓 `cargo fmt`；只格式化本任务实际修改的 Rust 文件。
- 验证顺序遵循经验文件：先前端 `npm run ui:build`，再 `cargo check`，二者不并行。
- 鼠标、快捷键、文字注入、字幕拖动和 OBS 交互由用户验证，任务完成时必须提供具体步骤。

## 1.1 固化的窗口生命周期与性能口径

- 主窗口销毁后继续：听写、实时字幕、字幕翻译、OBS 输出、文件转写、文稿对齐计算、模型对比中已启动的任务。
- 主窗口销毁后停止：依赖可见界面的音频试听、波形预览和尚未提交的表单草稿；关闭前须提示或安全停止占用音频设备的临时试听。
- 重建顺序固定为：拉取快照 → 订阅事件 → 再拉快照并以 revision 校正；事件须带 revision、domain、eventType 和可选 sessionId。
- 资源口径统一使用 `scripts/测量进程内存.ps1`，统计 `SayIt.exe` 根进程及全部 WebView2 子进程；托盘样本固定等待 60 秒，CPU 固定采样 5 秒。
- 最终目标暂锁定为：托盘总工作集相对当前托盘基线下降至少 40%；托盘空闲 CPU 不高于当前基线；主窗口恢复与首次快捷键响应不得比当前基线增加超过 100ms。真实基线待用户采集，采集前不判定通过。
