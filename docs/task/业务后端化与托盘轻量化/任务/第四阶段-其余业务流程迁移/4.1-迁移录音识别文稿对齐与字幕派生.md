# 4.1 迁移录音识别、文稿对齐与字幕派生

## 任务状态

- 状态：待开始
- 所属阶段：第四阶段-其余业务流程迁移
- 前置任务：1.2、2.2
- 关联任务：3.3、5.1
- 是否可以并行：可以；在 2.2 完成后可与 3.1/3.2 并行，但公共状态和事件文件需协调

## 任务目标

把录音文件识别、文稿对齐的任务生命周期、识别缓存、字幕 cue 派生、结果导出和取消语义迁到 Rust。主窗口销毁后任务继续运行，重建后可恢复进度和结果；前端保留文件选择展示和字幕编辑器的高频交互。

## 当前情况

- Rust 已实现文件信息、转写任务、取消、文本对齐算法、文件保存和音频解码。
- `features/transcription/controller.ts` 仍管理 active job、事件监听、任务阶段、对齐识别缓存和先识别再对齐流程。
- `features/transcription/subtitles.ts` 约 400 行，负责按词切字幕、合并、时间线规整、可编辑 cue 转换、SRT/纯文本导出和时间格式解析。
- `useTranscriptionStore.ts` 混合持久领域结果与页面 tab/resultView/saveMessage 等 UI 状态。
- `SubtitleEditor.tsx` 已拆分但仍含拖拽、缩放、播放头和编辑工作副本；这些高频操作不适合逐帧经 IPC。
- 既有《录音识别与文稿对齐》计划仍有用户端到端验证项，应在本任务前确认基线。

## 实施方案

- 建立后端 `TranscriptionService`，统一创建 job、进度、取消、结果缓存、错误和恢复快照；供应商调用走 2.2 能力。
- 将“识别后自动对齐”建模为一个复合 job，子任务状态不暴露供应商专属细节。
- 把字幕切分、合并、时间线规整、SRT/纯文本导出迁为 Rust 纯函数并补测试。
- 后端保存权威识别结果和对齐结果；前端打开编辑器时取得可编辑文档快照。
- 字幕编辑器在前端维护当前交互草稿，拖拽/输入过程中不逐帧调用 Rust；在操作提交、保存或切换文档时发送批量 patch，并由后端再次校验时间线。
- 页面 tab、展开项、当前 resultView、即时保存提示等纯 UI 状态留前端。

## 执行步骤

1. 完成既有录音识别/对齐人工基线验证，记录真实文件格式、长任务、取消和导出行为。
2. 定义转写/对齐 job 状态、快照、进度事件和错误结构，统一现有 `transcription-event` 的字符串 stage。
3. 实现后端服务，迁移 active job、target、pending script、识别缓存和复合任务编排；窗口监听者不存在时仍保存最新状态。
4. 将 `subtitles.ts` 中不依赖 DOM 的 cue 生成、时间规整、格式化和导出逐项迁到 Rust，使用当前边界值建立测试。
5. 设计编辑文档 patch 协议，至少支持文本、开始/结束时间、插入、删除、排序；后端拒绝无效时间线并返回规范化结果。
6. 修改前端 store 为 job/document 快照缓存；controller 缩为选择文件、发送命令和页面导航。
7. 验证任务运行中关闭/重开主窗口、取消后迟到事件、相同文件+参数缓存命中、长字幕编辑和导出。
8. 用户执行字幕拖拽、缩放、播放、文本编辑、SRT 保存等鼠标交互回归。

## 涉及内容

### 新增文件

- `src-tauri/src/application/transcription/`：job 服务、复合对齐流程、文档模型和事件。
- `src-tauri/src/application/subtitle_document.rs` 或等价模块：cue 生成、规整、patch、导出和测试。

### 修改文件

- `src-tauri/src/commands/transcription.rs`：变为服务入口，保留现有命令兼容。
- `src-tauri/src/text_align/`、`providers/alibabacloud/transcription/`：原则上不改算法/协议，只加薄适配。
- `src-tauri/src/state.rs`、`persistence.rs`：保存必要 job/result 快照；大结果是否落独立文件在执行时按体积确认。
- `ui/src/features/transcription/controller.ts`、`subtitles.ts`：删除后端化逻辑或缩为 UI 适配。
- `ui/src/store/useTranscriptionStore.ts`：拆分领域快照与 UI 状态。
- `ui/src/components/transcription/SubtitleEditor.tsx` 及子模块：批量 patch 和规范化结果接入，交互算法保留。

### 可能废弃

- 前端 active job/event listener、pending align 数据和识别缓存。
- 前端纯业务 cue 派生与 SRT 导出函数；时间输入的展示格式函数可继续保留。

## 特殊风险与回退要求

- 编辑器高频交互不得因 IPC 变卡；只在提交边界同步。
- 大型识别结果不应反复写进主 JSON 导致阻塞；是否使用独立缓存文件需测量后决定。
- 关闭窗口时后台 job 的继续/停止语义必须遵守 1.1 决定。
