# 4.2 迁移模型对比与音频调试编排

## 任务状态

- 状态：已完成
- 所属阶段：第四阶段-其余业务流程迁移
- 前置任务：1.2、2.2、3.1
- 关联任务：5.1
- 是否可以并行：可以；可与 3.2 或 4.1 并行，音频协调器与公共事件改动需协调

## 任务目标

把模型对比的多会话调度、录音/上传 PCM 管理、识别结果汇总，以及音频调试的录音、DSP 重处理和播放会话迁到 Rust。前端保留模型网格、波形 Canvas、参数控件和播放进度显示，不再缓存完整音频或驱动后台计时器。

## 当前情况

- `features/compare/controller.ts` 约 400 行，在前端同时维护多个实时/文件模型 session、PCM 索引、100ms feeder timer、录音块、HTMLAudioElement 和完成判断。
- `features/audio/lab.ts` 在前端缓存 raw/processed Float32Array，调用 Rust 离线 DSP 后自行编码 WAV、播放并绘制波形。
- Rust 已有 `encode_mono_wav_file`、`decode_audio_file_pcm`、`process_audio_offline`、CPAL 采集和 DSP 实现，但缺少对比与音频实验室的应用级会话。
- `useCompareStore` 的偏好在 1.2 后应已由 Rust 持久化；`useAudioStore` 当前主要是 UI 状态。
- 这些功能通常要求主窗口可见，但迁移后关闭窗口行为仍需符合 1.1 契约，不能靠 `beforeunload` 碰巧中断。

## 实施方案

### 模型对比

- Rust `CompareService` 根据后端模型目录构建 cell，统一打开实时 session 或文件 job。
- 录音模式由音频协调器一次采集并扇出给多个实时连接器；文件模型在停止后使用同一份音频文件/缓存。
- 上传模式由 Rust 解码一次，按固定节奏送实时模型，同时提交文件模型；进度和每格状态通过聚合事件输出。
- 关闭、取消和失败必须终止所有子任务并等待资源清理，不允许遗留 session。

### 音频调试

- Rust 持有 raw/processed PCM 和 DSP 参数，参数变化使用可取消 debounce 重处理。
- 播放改为后端原生会话或可控的临时音频资源。若原生播放依赖不合适，可以保留前端播放，但完整 PCM 不应长期存在于多个 JS 副本；具体方案执行时以性能测试决定。
- 前端只获取降采样波形点、统计值、播放进度和可播放标记。

## 执行步骤

1. 为模型对比建立测试矩阵：录音/上传、实时/文件混合、部分失败、全部失败、用户停止、页面关闭和迟到事件。
2. 实现 `CompareService`，复用 3.1 音频协调器与 2.2 provider 能力；把多 cell session 和完成判断迁入 Rust。
3. 将上传 PCM 解码、节奏投喂和文件临时资源统一放到后端，避免 base64/Float32Array 在 IPC 两侧复制。
4. 修改前端 compare controller/store，只发送用户意图并应用 cell 状态快照。
5. 实现 `AudioLabService`，迁移 capture、raw/processed 缓冲、DSP debounce、统计与资源释放。
6. 评估并确定播放实现；无论使用原生或临时资源，都必须支持停止、切换、窗口关闭和错误恢复。
7. 为 Canvas 提供固定数量的波形摘要点，不传完整 PCM；保持当前绘制效果。
8. 验证长文件、最大 4 行模型、连续开始停止、设备切换和关闭窗口后的资源释放。

## 涉及内容

### 新增文件

- `src-tauri/src/application/compare.rs` 或模块目录：多模型任务调度与聚合状态。
- `src-tauri/src/application/audio_lab.rs`：调试录音、处理、播放和波形摘要。
- 对应 fake provider/audio 测试。

### 修改文件

- `src-tauri/src/commands/compare.rs`、`commands/audio.rs`：变为应用服务入口。
- `src-tauri/src/audio_dsp.rs`、`audio_prep.rs`：原则上只暴露必要纯函数，不改算法。
- `src-tauri/src/desktop/backend_mic.rs`：通过音频协调器提供对比/调试 owner。
- `ui/src/features/compare/controller.ts`、`useCompareStore.ts`：删除会话和 PCM 权威。
- `ui/src/features/audio/lab.ts`、`useAudioStore.ts`、`views/AudioView.tsx`：改为命令/状态/波形摘要适配。

### 可能废弃

- 前端 `feederTimer`、activeCells、recordedChunks、pcmSamples、HTMLAudioElement 会话。
- 前端 WAV 编码和完整 PCM base64 转换路径；若字幕编辑器仍使用相关工具，只删除无消费部分。
- 后端 raw chunk 向主窗口广播：确认无消费者后在 5.1 删除。

## 特殊风险与回退要求

- 多模型任务必须有统一取消树，不能只移除前端监听器。
- 完整 PCM 的跨 IPC 复制是本任务明确要消除的问题，不能以换一种编码继续大量传输。
- 原生播放若无法保持设备兼容或无明显收益，允许保留受控前端播放；必须在执行记录说明原因，这不影响其他后端化目标。

## 验收标准

- [ ] 模型对比录音/上传、实时/文件混合和最大单元格数量均正常。
- [ ] 子任务成功、失败、取消和迟到事件有自动测试，不遗留 ASR/转写 session。
- [ ] 前端不再持有完整录音 PCM、投喂 timer 或后台任务权威。
- [ ] 音频调试参数变化、录音停止、重处理、原/处理后播放行为保持一致。
- [ ] Canvas 只接收波形摘要且视觉无明显退化。
- [ ] 关闭窗口后的继续/停止和资源释放符合 1.1 契约。
- [ ] 自动检查通过，用户完成模型对比和音频调试交互测试。
- [ ] 任务总览已同步更新。
