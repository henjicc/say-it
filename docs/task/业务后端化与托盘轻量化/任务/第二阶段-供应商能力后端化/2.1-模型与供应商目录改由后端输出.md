# 2.1 模型与供应商目录改由后端输出

## 任务状态

- 状态：已完成（人工页面核对统一纳入最终验收）
- 所属阶段：第二阶段-供应商能力后端化
- 前置任务：1.1
- 关联任务：2.2、3.1、3.2、4.1、4.2
- 是否可以并行：可以；可与 1.2 并行，公共契约和 `main.rs` 改动需提前协调

## 任务目标

让 Rust 成为模型元数据、场景选项、供应商能力和配置描述的唯一输出端。前端不再直接导入 `shared/asr-models.json`、推断协议族或自行选择供应商，只消费后端目录并渲染。

## 当前情况

- `shared/asr-models.json` 已是前后端共享模型注册表，后端 `providers/registry.rs` 和前端 `features/asr/modelRegistry.ts` 各自解析一遍。
- 前端 `modelOptions.ts`、`compare/models.ts` 和多个 store 仍负责默认模型、场景选项、模型类别及能力判断。
- `useProviderStore.ts` 保存 profile、defaults 和 override，并在前端实现 `effective()`、`optionsFor()`、`labelFor()`。
- 当前新增同协议模型已能只改注册表，但新增供应商仍可能需要修改前端判断和专属设置页面。
- 现有 `ProviderListItem` 只返回基础 profile 信息，没有稳定的模型目录、配置字段描述和可用操作描述。

## 实施方案

1. 保留 `shared/asr-models.json` 作为仓库可维护数据源，但运行时只允许 Rust 解析和校验。
2. 新增后端目录响应，至少包含：模型 ID/显示名、供应商 ID、类别、协议、可用场景、能力、默认标记；供应商 profile、默认值、状态、普通配置字段描述和可用专属操作。
3. 将默认模型、场景过滤、供应商有效选择和历史未知模型兜底统一放入 Rust。
4. 前端提供薄查询函数，输入后端目录并返回下拉项；这些函数不能重新编码协议知识。
5. 配置描述只覆盖通用字段（如 API Key、Base URL、普通布尔/枚举）；热词等复杂能力允许保留专属面板，通过后端声明的 capability/action 决定是否显示。

## 执行步骤

1. 审核注册表所有字段和现有消费方，区分领域字段与纯展示字段；删除字段前需证明无消费方并更新操作手册。
2. 在 Rust 定义 `ModelCatalog`、`ProviderCatalogItem`、`ProviderConfigField` 和可用 action/capability 类型，添加注册表完整性测试。
3. 实现目录查询命令或合并进 1.1 的应用快照，确保目录版本可用于前端缓存失效。
4. 将 `defaultRealtimeModel`、`defaultFileModel`、`optionsForScene`、`modelKind`、供应商有效选择等逻辑改为后端输出。
5. 修改前端 store 与模型下拉消费方，移除对 `~shared/asr-models.json` 的直接 import 和协议字符串判断。
6. 保持现有下拉顺序、标签、默认选中值和未知历史模型兜底行为，逐页面对比听写、字幕、转写和对比选项。
7. 更新 `docs/rules/新增供应商与模型操作手册.md`，明确前端不再需要登记模型知识。

## 涉及内容

### 新增文件

- `src-tauri/src/application/catalog.rs` 或执行时确认的目录服务模块。
- 目录契约和注册表完整性测试。

### 修改文件

- `src-tauri/src/providers/registry.rs`、`providers/mod.rs`：输出完整模型/供应商目录。
- `src-tauri/src/commands/session.rs` 或新的薄目录命令文件：暴露目录。
- `shared/asr-models.json`：原则上不改数据，只在契约需要时补充真实字段。
- `ui/src/features/asr/modelRegistry.ts`、`modelOptions.ts`、`features/compare/models.ts`：改为消费后端目录。
- `ui/src/store/useProviderStore.ts`：去除领域选择逻辑，保留目录快照和 UI override 意图；override 是否属于领域配置需按 1.1 契约确定。
- 各模型下拉页面：改用后端目录，视觉和选项不变。
- `docs/rules/新增供应商与模型操作手册.md`：同步新增流程。

### 可能废弃

- 前端直接解析 `shared/asr-models.json` 的代码。
- `modelOptions.ts` 中协议/供应商能力判断；可保留同名薄函数以减少一次性改动，最终在 5.1 清理。
