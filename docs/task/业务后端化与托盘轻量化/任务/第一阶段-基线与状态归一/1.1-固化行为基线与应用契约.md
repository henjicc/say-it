# 1.1 固化行为基线与应用契约

## 任务状态

- 状态：已完成（人工回归与性能基线按用户决定统一纳入最终验收）
- 所属阶段：第一阶段-基线与状态归一
- 前置任务：无
- 关联任务：全部后续任务
- 是否可以并行：否；本任务完成后 1.2 与 2.1 可以并行

## 任务目标

在迁移任何业务前，把当前版本的功能行为、关闭窗口语义、内存/CPU/唤起时延和前后端消息结构固化为可执行基线；同时定义后续 Rust 应用服务共同使用的状态快照、领域事件和测试替身边界。完成后，其他 AI 无需重新推断“当前行为是什么、迁移后怎样算等价”。

## 当前情况

- `ui/src/hooks/useTauriBridge.ts` 同时分发 ASR、转写、听写快捷键、字幕快捷键、翻译和悬浮窗按键事件，是当前主窗口运行时入口。
- `ui/src/lib/tauri.ts` 维护命令/事件字符串，但没有集中定义 payload schema、版本或事件先后约束。
- `RuntimeState` 只覆盖部分后端状态；听写、字幕、转写、对比的核心运行状态仍是 TypeScript 模块变量。
- `docs/task/大文件拆分与遗留问题修复/` 与 `docs/task/录音识别与文稿对齐/` 仍有待用户实测项。迁移前必须把这些结果记录为基线，不能把旧问题误判为迁移回归。
- 当前内存截图只有单点数据，尚无统一进程树采样方式，也没有主窗口恢复和快捷键响应的时延基线。

## 实施方案

建立一份面向代码的契约，而不是另写大而空的架构文档：

1. 为核心命令/事件补 Rust 可序列化类型，统一 `camelCase`，前端类型从对应契约人工镜像并由契约测试校验关键字段。
2. 定义 `AppSnapshot`（配置摘要 + 各领域运行状态 + 递增 revision）与统一领域事件信封，明确“先拉快照、再订阅、再校正 revision”的重建流程。
3. 为未来应用服务定义最小端口：供应商、音频输入、文本注入、指示器、持久化和事件发布。只定义真实需要的切面，不提前实现全套框架。
4. 建立 Windows 进程树采样方式，记录前台、关闭到托盘 60 秒、一次听写后、实时字幕运行中四类状态的工作集、私有字节、CPU 和进程数。
5. 将关闭主窗口时各业务应继续还是停止的语义写入本任务执行记录和 `重要记录.md`。

## 执行步骤

1. 阅读两个既有待验证计划，整理与本迁移相关的人工回归清单；由用户完成快捷键、注入、字幕、转写、对齐和字幕编辑器验证，记录已知异常。
2. 枚举 `CMD`、`EVT` 和对应 Rust 命令/emit，整理仍由字符串和 `never` 强转表达的 payload；优先为后续会跨窗口重建的状态建立 Rust 类型。
3. 在后端新增应用契约模块，定义快照 revision、领域状态枚举、错误结构和事件信封；为序列化字段添加单元测试。
4. 增加只读快照命令的最小骨架，初期可以只汇总现有 Rust 状态和“尚由前端负责”的占位信息，不迁移任何领域权威。
5. 建立可重复的 PowerShell 采样脚本或等价测试工具：从 `SayIt.exe` 根 PID 递归统计子进程，输出 JSON/CSV，采样条件和等待时间固定。
6. 记录主窗口显示、隐藏、恢复和快捷键指示器的基线时延；涉及实际点击和快捷键的操作交给用户执行。
7. 在执行记录中锁定性能目标和关闭窗口生命周期语义，并同步更新实施方案与重要记录。
