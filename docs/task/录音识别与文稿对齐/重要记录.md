# 重要记录

## 记录 001：文稿对齐作为「录音识别」页内 Tab，而非独立导航项

- 日期：2026-07-03
- 状态：已确定
- 背景：文稿对齐与录音转写共用「文件上传 → Fun-ASR 识别 → 词级时间戳」整条链路，仅结果加工方式不同。
- 最终决定：左侧导航只新增一个「录音识别」，页内用 Tabs 分「录音转写」与「文稿对齐」。
- 选择原因：链路复用最大化、导航不膨胀；两 Tab 状态在 store 中相互独立，互不干扰。
- 影响任务：2.1、3.2。

## 记录 002：文件上传采用百炼临时 OSS（getPolicy 直传），接受官方“非生产”限制

- 日期：2026-07-03
- 状态：已确定
- 背景：官方注明临时 URL 48 小时有效、凭证接口限流 100 QPS、不建议生产环境；替代方案是用户自备正式 OSS。
- 最终决定：使用临时 OSS。识别任务提交在上传后立即进行，48 小时有效期完全够用；桌面单用户场景触发不了限流。
- 选择原因：零额外配置（复用同一 API Key），符合“个人桌面工具”定位；正式 OSS 需要用户额外开通与配置，体验差。
- 影响任务：1.1、1.2。
- 附加约束（写死在实现里）：调用模型时必须加 `X-DashScope-OssResourceResolve: enable`；上传时 `model` 参数必须与识别用模型一致；同一 API Key。

## 记录 003：实时识别热词词表暂不自动复用到录音识别

- 日期：2026-07-03
- 状态：待确认
- 背景：现有热词词表以 `target_model=fun-asr-realtime` 创建（`customization.rs`），而录音识别模型为 `fun-asr`；文档未明确词表能否跨 target_model 使用，且 `protocol.rs` 注释明确提示 target_model 不一致会静默不生效。
- 最终决定（暂定）：录音识别参数面板的 `vocabulary_id` 默认空、手动填写；不自动带入实时识别的词表 ID。
- 选择原因：避免“看起来配了热词实际不生效”的静默错误。
- 待确认事项：实测词表跨模型是否生效；若需要，后续可增任务“为 fun-asr 创建并同步第二份词表”。
- 影响任务：2.2。

## 记录 004：新增 tauri-plugin-dialog 依赖；写文件走自建命令不引入 tauri-plugin-fs

- 日期：2026-07-03
- 状态：已确定
- 背景：文件选择/保存对话框无法用现有依赖合理实现；导出 SRT 需要写文件。
- 最终决定：新增官方插件 `tauri-plugin-dialog`（Rust + npm 两侧）；写文件用一个简单的后端命令 `save_text_file`，不引入 `tauri-plugin-fs`。
- 选择原因：dialog 是官方维护、体积小、无替代；fs 插件对“写一个文本文件”过重，自建命令更可控。
- 影响任务：2.1、2.3、3.2。

## 记录 005：录音识别沿用 dashscope.aliyuncs.com 通用域名（不引入 WorkspaceId 专属域名）

- 日期：2026-07-03
- 状态：待确认
- 背景：官方文档主推 `{WorkspaceId}.cn-beijing.maas.aliyuncs.com` 新域名，但需要用户提供 Workspace ID；现有实时识别与热词接口均使用通用域名。
- 最终决定（暂定）：沿用通用域名，与现有代码一致、用户零配置；请求体始终携带 `parameters`（即使为空对象）以规避文档提到的新域名坑，通用域名下同样安全。
- 待确认事项：1.2 联调时验证通用域名 + `oss://` URL + `X-DashScope-OssResourceResolve: enable` 组合可用；若不可用，回退方案为增加 Workspace ID 配置项并切换新域名（届时更新本记录与实施方案）。
- 影响任务：1.2。
