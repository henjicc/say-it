# 录音识别与文稿对齐 · 实施方案

> 本文记录相对稳定的整体方案与项目基础信息。任务进度见 [00-任务总览.md](00-任务总览.md)。

## 一、项目现状（执行任何任务前先了解）

### 技术栈与命令

- Tauri 2 + Rust 2021 后端（`src-tauri/src`），React 19 + TypeScript + Vite 6 + Tailwind CSS 4 前端（`ui/src`），Windows 为主要目标。
- 开发：`npm run tauri:dev`（完整应用）/ `npm run ui:dev`（仅前端，端口固定 5155）。
- 检查：`npx tsc --noEmit`（前端类型检查，在仓库根目录执行）、`cargo check`（在 `src-tauri` 目录执行）。无 lint / 自动化测试脚本。

### 与本任务相关的现有实现

- **供应商体系**：`src-tauri/src/providers/mod.rs` 定义唯一供应商 `funasr`（kind `alibabacloud-funasr`），配置存于 `ProviderProfile.config`（JSON，含 `apiKey`、`vocabularyId`、`languageHints` 等）。`normalize_settings` 会强制 `capabilities = ["asr"]` 并只保留 `funasr` profile。API Key 读取方式见 `src-tauri/src/commands/funasr.rs` 的 `funasr_credentials`。
- **实时识别**：`src-tauri/src/commands/asr.rs` 通过 WebSocket（`providers/alibabacloud/urls.rs`、`protocol.rs`）流式调用 `fun-asr-realtime`，与本任务的 HTTP 异步接口互不影响。
- **HTTP 调用范式**：`providers/alibabacloud/customization.rs` 的 `post_customization` 展示了 reqwest + Bearer 鉴权 + 错误信息提取的既有写法，新 HTTP 模块照此风格实现。
- **命令注册**：Tauri command 定义在 `src-tauri/src/commands/*.rs`，在 `src-tauri/src/main.rs` 的 `invoke_handler` 中注册；前端命令名集中在 `ui/src/lib/tauri.ts` 的 `CMD`，事件名在 `EVT`。
- **前端结构**：导航项在 `ui/src/components/shell/navConfig.tsx`；视图键 `ViewKey` 在 `ui/src/store/useUiStore.ts`；视图组件映射在 `ui/src/App.tsx` 的 `VIEWS`。现有 UI 组件：`Collapse`（折叠面板）、`Tabs`、`Button`、`Field/CheckField`、`Input`、`Slider`、`Modal` 等，位于 `ui/src/components/ui/`。参数折叠面板的样式范例见 `ui/src/views/SettingsProviderPanel.tsx`。
- **事件桥**：后端向前端推送用 `app.emit`（参考 `commands/audio.rs` 的 `emit_asr_stream_event`），前端用 `ui/src/lib/tauri.ts` 的 `on()` 监听。

### 现状空缺

- 无任何文件上传、HTTP 异步任务轮询、文件选择对话框能力。
- `reqwest` 未启用 `multipart` feature（`src-tauri/Cargo.toml`），表单上传需要补。
- 无 `tauri-plugin-dialog`（文件选择/保存对话框需要）。

## 二、最终目标

1. 左侧导航新增「录音识别」页面，页内两个 Tab：**录音转写** 与 **文稿对齐**。
2. 录音转写：选择/拖入本地音视频文件 → 上传临时 OSS → 提交 `fun-asr` 异步任务 → 轮询 → 展示结果（纯文本 / 带时间戳字幕两种视图）→ 复制 / 导出 SRT。
3. 参数默认折叠（模型版本、语种提示、热词 ID、说话人分离等），有合理默认值，普通用户不需要展开。
4. 文稿对齐：音频 + 一行一句的文稿 → 识别 → 词级时间戳与文稿对齐 → 生成文本完全等于文稿、时间轴精确的字幕 → 导出 SRT。

## 三、外部接口事实（已通过官方文档核实，2026-07-03）

### 3.1 临时 OSS 上传（两步）

复用 `funasr` profile 的 `apiKey`。文档：https://help.aliyun.com/zh/model-studio/get-temporary-file-url

**第一步 获取上传凭证**：
`GET https://dashscope.aliyuncs.com/api/v1/uploads?action=getPolicy&model=fun-asr`，请求头 `Authorization: Bearer {apiKey}`。返回 `data` 对象字段：`policy`、`signature`、`upload_dir`、`upload_host`、`oss_access_key_id`、`x_oss_object_acl`、`x_oss_forbid_overwrite`、`max_file_size_mb`、`expire_in_seconds`（凭证 300 秒有效）。

**第二步 表单直传 OSS**：
`POST {upload_host}`，`multipart/form-data`，字段顺序：`OSSAccessKeyId`、`Signature`、`policy`、`x-oss-object-acl`、`x-oss-forbid-overwrite`、`key`（= `{upload_dir}/{文件名}`）、`success_action_status`（`200`）、`file`（必须最后）。成功返回 HTTP 200 空 body。

**结果 URL**：`oss://{upload_dir}/{文件名}`，48 小时有效。**硬性约束**：后续模型调用必须（a）加请求头 `X-DashScope-OssResourceResolve: enable`；（b）与上传时的 `model` 一致；（c）与上传时同一 API Key 主账号。上传凭证接口限流 100 QPS。

### 3.2 Fun-ASR 录音文件识别（异步，提交 + 轮询）

文档：https://help.aliyun.com/zh/model-studio/fun-asr-recorded-speech-recognition-http-api （不使用 WorkspaceId 专属域名，沿用 `dashscope.aliyuncs.com` 通用域名，与现有实时识别、热词接口一致）

**提交**：`POST https://dashscope.aliyuncs.com/api/v1/services/audio/asr/transcription`
请求头：`Authorization: Bearer {apiKey}`、`Content-Type: application/json`、`X-DashScope-Async: enable`、（oss:// URL 时）`X-DashScope-OssResourceResolve: enable`。
请求体：`model`（`fun-asr` / `fun-asr-2025-11-07` / `fun-asr-2025-08-25` / `fun-asr-mtl` / `fun-asr-mtl-2025-08-25`）、`input.file_urls`（数组，单次 1 个）、`parameters`（可选：`vocabulary_id`、`channel_id`、`diarization_enabled`、`speaker_count`、`language_hints`、`special_word_filter`）。
返回：`output.task_id`、`output.task_status = PENDING`。

**轮询**：`GET https://dashscope.aliyuncs.com/api/v1/tasks/{task_id}`，头 `Authorization`。
`output.task_status`：`PENDING` / `RUNNING` / `SUCCEEDED` / `FAILED`。成功时 `output.results[0].transcription_url`（结果 JSON 下载链接，24 小时有效；子任务失败时该元素含 `code`/`message`，`subtask_status=FAILED`）。注意：整体 `SUCCEEDED` 不代表子任务成功，须检查 `subtask_status`。

**结果 JSON 结构**（从 `transcription_url` 下载，无需鉴权）：
`properties`（`original_duration_in_milliseconds` 等）+ `transcripts[]`，每个含 `channel_id`、`text`（全文）、`sentences[]`（`begin_time`/`end_time`/`text`/`sentence_id`/可选 `speaker_id`/`words[]`），`words[]` 含 `begin_time`/`end_time`/`text`/`punctuation`（毫秒时间戳）。

### 3.3 音频规格（fun-asr / fun-asr-mtl 系列，非实时）

- 格式：aac、amr、avi、flac、flv、m4a、mkv、mov、mp3、mp4、mpeg、ogg、opus、wav、webm、wma、wmv（含视频容器，直接支持）。
- 采样率任意；文件 ≤ 2 GB；时长 ≤ 12 小时（启用说话人分离建议 ≤ 2 小时）。

## 四、总体方案

### 4.1 后端（src-tauri）

新增模块，遵循现有依赖方向（providers 提供纯协议能力，commands 编排并 emit 事件）：

- `providers/alibabacloud/uploads.rs`：getPolicy + multipart 直传，输出 `oss://` URL。
- `providers/alibabacloud/transcription.rs`：提交任务、查询任务、下载并解析结果 JSON 为结构化类型（`TranscriptionResult` → `sentences[]` → `words[]`）。
- `commands/transcription.rs`：`transcription_start` 命令接收文件路径 + 参数，spawn 异步任务串起「上传 → 提交 → 轮询 → 下载解析」，各阶段通过事件 `transcription-event`（`stage`: `uploading` / `submitted` / `polling` / `completed` / `error`）推送前端；提供取消命令（只终止本地轮询）。
- 文稿对齐纯算法放 `src-tauri/src/text_align.rs`，由 `commands/transcription.rs` 暴露 `align_transcript` 同步命令（输入词级时间戳 + 文稿行，输出行级字幕 + 匹配率）。

轮询策略：首次延迟 2s，之后间隔 3~5s，上限 30 分钟超时报错。

### 4.2 前端（ui/src）

- `useUiStore` 增加 `ViewKey = "transcription"`，`navConfig.tsx` 增加「录音识别」导航项，`App.tsx` 注册 `TranscriptionView`。
- `views/TranscriptionView.tsx`：Tabs 切换「录音转写」/「文稿对齐」；状态放 `store/useTranscriptionStore.ts`（文件、参数、任务阶段、结果、对齐文稿），业务流程放 `features/transcription/controller.ts`。
- 参数区用现有 `Collapse` 组件默认收起，参数持久化到 `funasr` profile 的 `config`（新增 `transcription` 子对象，经现有 `update_provider_config` 命令保存）。
- 文件选择用 `tauri-plugin-dialog`；同时支持拖放（Tauri webview drag-drop 事件提供真实路径）。
- SRT 导出：前端生成 SRT 文本，用 dialog 的保存对话框 + 新增简单写文件命令（或 `tauri-plugin-fs`，取其一，倾向后端命令写文件、少一个插件）。

### 4.3 字幕组句策略（录音转写的“带时间戳字幕”视图）

结果 JSON 已含句级 `sentences[]`，直接以句子为字幕基础单元；对超长句（默认 > 30 个 CJK 字符宽度）用 `words[]` 二次切分：优先在词后带标点处断开，其次在相邻词时间间隙最大处断开。过短相邻句（< 1s 且合计不超长）可合并。该策略实现为纯函数，便于文稿对齐复用导出逻辑。

### 4.4 文稿对齐算法（核心思路）

问题本质是 forced alignment 的文本域近似：ASR 词序列（带时间戳）与用户文稿存在插入/删除/替换差异，需要鲁棒对齐。

1. **规整**：两侧文本 NFKC 归一、转小写、去标点与空白；按「CJK 单字 = 1 token、连续拉丁字母/数字 = 1 token」切分；保留 token → 原文稿行号的映射。ASR 侧每个 token 继承所属词的时间，多字词内部按字数线性内插细化时间。
2. **对齐**：锚点 + 分治。先用哈希找两侧唯一匹配的 n-gram（n=5 起，找不到降到 3）作为锚点，把长序列切成小段；段内用 Needleman-Wunsch（match=+2、mismatch=-1、gap=-1）做经典 DP 对齐。整体复杂度可控，几万字文稿可行。
3. **行级时间**：每行的 begin = 该行第一个成功匹配 token 的时间，end = 最后一个匹配 token 的结束时间；整行无匹配时按相邻已匹配行的时间线性内插；最后强制时间轴单调、不重叠、且每条字幕不短于最小时长（如 300ms）。
4. **质量反馈**：输出每行匹配率（匹配 token / 行 token），前端对低于阈值（如 60%）的行标黄提示，便于用户发现文稿与音频严重不符的段落。
5. **已知局限**（如实展示，不隐藏）：数字/符号读法差异（“2024” vs “二零二四”）会降低局部匹配率，靠 gap 容忍与内插兜底，首版不做数字归一化。

### 4.5 阶段划分与关系

- 第一阶段（后端能力）与第二阶段的 2.1（页面框架）可并行；2.2 依赖两者汇合。
- 第三阶段复用 1.2 的词级结果与 2.2 的执行流程，3.1（算法）可在 1.2 完成后先行。
- 3.3 为端到端验证收尾，两条功能线都完成后执行。

## 六、风险与回退

- **oss:// URL + 通用域名 + 异步转写的兼容性**：文档主推 WorkspaceId 域名，通用域名 + `X-DashScope-OssResourceResolve: enable` 理论可行但需 1.2 尽早真机验证；若失败，备选方案是走 WorkspaceId 域名（需要用户提供 Workspace ID，配置项预留）。
- **热词跨模型**：现有词表 `target_model=fun-asr-realtime`，对 `fun-asr` 可能无效（见 重要记录 003）；首版参数面板的 `vocabulary_id` 默认为空、手动填写，不自动复用。
- **大文件上传**：凭证仅 300 秒有效，上传须在获取凭证后立即进行；2GB 上限由前端在选择文件后校验提示。
- **回退原则**：全部为新增模块/页面，不触碰实时识别路径；任一任务失败可整体回退对应提交，不影响既有功能。

## 七、全局开发约束

## 八、最终验收条件

1. 在「录音识别」页选择一个本地 mp3/wav/mp4 文件，点击开始后可看到上传/识别进度，最终得到纯文本与带时间戳字幕两种结果，可复制、可导出合法 SRT 文件。
2. 参数面板默认折叠，展开可改模型版本、语种、说话人分离等并持久化。
3. 「文稿对齐」Tab 提供音频 + 逐行文稿，产出的字幕文本与文稿逐行一致、时间轴在人工抽查处与音频吻合，低匹配行有可见提示，可导出 SRT。
4. 现有语音输入、实时字幕功能回归正常（至少人工冒烟一次）。
