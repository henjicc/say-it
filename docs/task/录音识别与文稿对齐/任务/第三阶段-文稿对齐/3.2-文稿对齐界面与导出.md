# 3.2 文稿对齐界面与导出

## 任务状态

- 状态：待验证（功能实现完成，待用户真实音频实测）
- 所属阶段：第三阶段-文稿对齐
- 前置任务：2.2（识别执行流程）、3.1（对齐命令）
- 关联任务：2.3（复用 SRT 导出与 cue 渲染）
- 是否可以并行：否

## 任务目标

「文稿对齐」Tab 完整可用：选择音频 + 粘贴/输入逐行文稿 → 执行（识别 + 对齐）→ 展示行级字幕结果（低匹配行高亮提示）→ 导出 SRT。

## 当前情况

- 2.1 已留出文稿对齐 Tab 占位；2.2 的 controller 已实现「上传 → 识别 → 结果」流程；3.1 已提供 `align_transcript` 命令。
- SRT 导出（`toSrt` + `save_text_file`）与字幕列表渲染在 2.3 已实现，可直接复用（`features/transcription/subtitles.ts`）。
- 文稿对齐与录音转写共用文件选择逻辑，但状态应相互独立（两个 Tab 各自的文件与结果互不干扰）——store 设计时注意分开或按 Tab 命名空间。

## 实施方案

- store 增加对齐侧状态：`alignFile`、`scriptText`、`alignStage`（复用识别阶段枚举，追加 `aligning`）、`alignedLines`、`alignError`。
- 界面（文稿对齐 Tab）：
  - 文件选择区（复用 2.1 的组件/逻辑）。
  - 文稿输入：多行 textarea，说明文案「一行一句，字幕文本将完全按此文稿输出」；显示行数统计；空行自动忽略。
  - 执行按钮：校验文件与文稿均就绪；流程 = 复用 controller 识别（参数沿用参数面板配置）→ 拿到 `TranscriptionResult` 后拍平 words → invoke `align_transcript` → 结果入 store。
  - 结果列表：行号、时间段、文稿文本；`match_ratio < 0.6` 或 `interpolated` 的行黄色高亮，悬浮/旁注说明「该行与音频匹配度低，时间为估算」；顶部显示整体平均匹配率。
  - 导出 SRT 按钮（复用 2.3 逻辑，默认文件名 = 音频名 + `.对齐.srt`）。
- 识别结果可缓存：同一文件已在本 Tab 识别过且参数未变时，重复执行只重新对齐不重新识别（简单实现：记住上次识别的文件路径 + 参数指纹）。

## 执行步骤

1. store 扩展对齐侧状态；controller 增加 `startAlignment()`（识别 → 拍平 words → 调用对齐命令）。
2. 实现 Tab 界面：文件区、文稿输入、执行与进度、结果列表（含低匹配高亮）、导出。
3. `npx tsc --noEmit` 通过；真实音频 + 对应文稿跑通一次。
4. 交给用户实测：
   - 正常对齐：音频 + 准确文稿，导出 SRT 在播放器中逐句核对时间；
   - 差异文稿：故意改错/增删几行，观察低匹配行提示是否合理；
   - 两个 Tab 状态互不干扰。

- 实际改动：Rust `text_align.rs` 改为仿射 gap 计分（Gotoh）并输出 `similarity`、`asr_word_begin/end`（14 个单测全绿，新增“音频多说时相似度下降”用例）；`subtitles.ts` 增加 `buildOptimizedAlignCues`/`shouldUseAsrText`（阈值 0.4）；store 增加 `alignResultView`、`alignOptimizedCues`；面板增加视图切换与修正列表。
- 验证结果：`cargo test`（14/14）、`cargo check`、`npx tsc --noEmit`、`npm run ui:build` 通过；真实音频下修正效果待用户实测。
- 阈值决策记录：正常 ASR 错字率（10%~30%）下相似度仍 ≥0.7 不会误替换；改写约一半的行 ≈0.5 仍保文稿（用户文稿优先）；只有明显没照稿念的行才 <0.4 触发替换。音频里说了但文稿没写的内容（纯插入、两侧行都匹配良好）不会被插入修正版，与“文稿优先”定位一致。
