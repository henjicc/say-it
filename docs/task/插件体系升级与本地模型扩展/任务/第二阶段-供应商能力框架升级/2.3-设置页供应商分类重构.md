# 2.3 设置页供应商分类重构

## 任务状态

- 状态：待开始
- 所属阶段：第二阶段-供应商能力框架升级
- 前置任务：2.1（需要 ocr 能力类型与供应商列表输出）
- 关联任务：4.2（OCR 默认供应商选择的消费方）、3.2（本地模型供应商将出现在 ASR 分类）
- 是否可以并行：与第三阶段任务可并行

## 任务目标

设置页「密钥与识别」从单一「识别供应商」分类重构为「ASR 供应商」「OCR 供应商」「翻译供应商」三个分区：每个分区按能力过滤展示供应商（内置 + 插件），支持选择该能力的默认供应商；翻译插件（后端已支持）从此有了呈现入口。不改动 store 数据流、Tauri 命令与各供应商面板内部逻辑。

## 当前情况

- `ui/src/views/SettingsView.tsx`：标签 `{ key: "provider", label: "密钥与识别" }` → `SettingsProviderPanel.tsx`。
- `SettingsProviderPanel.tsx`：单一 `SettingsSection title="识别供应商"`；阿里云百炼面板硬编码（API Key 自动保存、热词 `FunAsrHotwordsPanel`、高级参数），插件供应商按 `kind.startsWith("plugin:")` 过滤后统一渲染 `PluginProviderConfig`（configFields 驱动 + actions 按钮 + 热词）。
- 供应商数据：`ui/src/store/useProviderStore.ts`（profiles 含 `capabilities` 数组、defaults）；默认供应商切换命令后端已有（`set_default_provider`，数据驱动，2.1 后支持 `ocr`）。
- 一个供应商可有多能力（阿里云百炼 = asr + llm + translation）：分类是**按能力过滤的视图**，同一供应商可出现在多个分区（配置面板本体只render一次还是每区一份，见实施方案）。
- LLM 供应商已有独立面板 `SettingsLlmPanel.tsx`，不在本任务范围。

## 实施方案

1. 「密钥与识别」页内改为三个 `SettingsSection`：「ASR 供应商」「OCR 供应商」「翻译供应商」，分别过滤 `capabilities` 含 `asr` / `ocr` / `translation` 的启用供应商。
2. **多能力供应商的呈现策略**：完整配置面板（密钥、热词、高级参数）只出现在其主能力分区（阿里云百炼 → ASR 分区）；其他分区里同一供应商只显示一行条目（名称 + 已配置状态 + "配置入口在 ASR 分区"跳转/提示），避免同一份密钥表单渲染多份造成状态打架。执行时若发现更自然的现有组件模式可调整，写入执行记录。
3. **默认供应商选择**：每个分区顶部一个默认供应商下拉（数据来自 defaults + 该能力的供应商列表，调用现有 `set_default_provider` 命令）。ASR 分区注意与现有"模型下拉"（听写/字幕页）的关系——本任务只做供应商默认值，不动模型选择逻辑。
4. OCR 分区初始只有「Windows 系统 OCR」（2.1 的内置 profile，无配置项，显示为不可配置条目）；4.1 完成后 PP-OCR 模型包供应商自动出现，无需再改本页。
5. 复用 `SettingsSection` / `Collapse` / `Field` 等既有组件与令牌，不新增样式硬编码。

## 执行步骤

1. `useProviderStore.ts` 增加 defaults 读取与设置 action（若尚未暴露 `set_default_provider` 调用，则补命令封装；执行前先确认现状）。
2. 重构 `SettingsProviderPanel.tsx`：拆出 `ProviderSectionForCapability`（能力过滤 + 默认下拉 + 条目渲染）组件；阿里云面板与 `PluginProviderConfig` 保持原实现，仅挪入对应分区。
3. 翻译分区验证：安装一个声明 translation 的测试插件（或用现有测试插件改 manifest），确认其出现在翻译分区且可配置。
4. 验证：`npx tsc --noEmit`、`npm run ui:build`；`npm run tauri:dev` 人工回归——API Key 保存、热词、插件动作按钮、默认供应商切换后持久化（重启软件仍生效）。
