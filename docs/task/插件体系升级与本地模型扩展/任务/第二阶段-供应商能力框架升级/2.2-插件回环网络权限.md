# 2.2 插件回环网络权限

## 任务状态

- 状态：已完成
- 所属阶段：第二阶段-供应商能力框架升级
- 前置任务：无（若 2.1 未完成，先在本任务内完成 API v4 版本放宽，两任务后合并冲突很小）
- 关联任务：2.1（同属 v4 manifest 扩展）
- 是否可以并行：可与 1.1、2.1 并行

## 任务目标

v4 插件可声明 `localNetwork` 权限，声明后允许对回环地址（`127.0.0.1` / `localhost` / `[::1]`）使用明文 `http` / `ws` 协议，使插件能接入用户自建的本机推理服务（如 Fun-ASR-Nano / Qwen3-ASR 的 vLLM WebSocket 服务）。非回环主机的规则保持不变（仅 https/wss + allowedHosts 白名单）。

## 当前情况

- `src-tauri/src/providers/plugin_runtime.rs::parse_allowed_url`：统一 URL 校验入口，调用点两处——`http_request`（仅 `["https"]`，含重定向跟随处的二次校验，约 371/420 行）与 `websocket_open`（仅 `["wss"]`，约 456 行）；主机名需匹配 manifest `allowedHosts`（支持 `*.` 通配）。
- 权限白名单：`plugin.rs` 的 `ALLOWED_PERMISSIONS = ["network", "browserSession", "cookies"]`。
- 现状后果：`ws://127.0.0.1:PORT` 直接被"不允许的网络协议"拒绝，本机 GPU 服务无法接入。

## 实施方案

1. `ALLOWED_PERMISSIONS` 加 `localNetwork`（仅 v4 manifest 允许声明，v3 声明报错——与 2.1 的版本策略一致）。
2. `PluginRuntimeSpec` 携带该权限标记；`parse_allowed_url` 扩展：当插件具有 `localNetwork` 且 URL 主机是回环地址（`127.0.0.1`、`::1`、`localhost` 精确匹配）时，额外放行 `http` / `ws` 协议，且**回环主机不要求出现在 allowedHosts**（本机端口由用户自己掌控，逐端口白名单无意义）；非回环主机走原逻辑不变。
3. 明确拒绝面：`0.0.0.0`、局域网 IP、任何解析到回环的自定义域名一律不放行（只按字面主机名判断，不做 DNS 解析防止绕过——`localhost` 是唯一例外且属系统保留名）。
4. 安装/管理 UI 的权限展示（`PluginManagerPanel.tsx` 若已列权限）同步显示"本机网络"字样；若当前 UI 不展示权限列表则不新增，记录到执行记录。

## 执行步骤

1. `plugin.rs`：权限白名单 + v3/v4 差异校验 + 测试。
2. `plugin_runtime.rs`：`parse_allowed_url` 加回环分支；补测试——`ws://127.0.0.1:8000` 有权限放行/无权限拒绝、`http://192.168.1.5` 有权限仍拒绝、`https` 原有用例不回归。
3. 检查 `http_request` 重定向跟随处的二次 `parse_allowed_url` 调用使用相同 schemes 参数（回环 http 的重定向仍受同一规则约束）。
4. 验证：`cargo test`；如有条件用本机起一个简单 WebSocket 回显服务做一次真实插件调用（可选，写明是否执行）。

## 执行记录

### 2026-07-18 完成实施

- **权限白名单**：`ALLOWED_PERMISSIONS` 加 `localNetwork`；v3 manifest 声明该权限报“localNetwork 权限需要插件 API 版本 4”（2.1 已把版本校验放宽到 3..=4）。
- **URL 校验**：`parse_allowed_url` 重构——先判协议是否属于 `schemes` 或其明文变体（https→http、wss→ws），再取主机；回环主机（`127.0.0.1` / `localhost` / `[::1]`，字面精确匹配，不做 DNS 解析）且插件声明 `localNetwork` 时放行明文与加密协议，且**不要求出现在 allowedHosts**；其余主机走原白名单逻辑不变。`[::1]` 因 Url 序列化带方括号，按带括号形式匹配。
- **重定向**：`http_request` 重定向跟随处复用同一 `parse_allowed_url` + 相同 schemes 参数（原有代码即如此），回环 http 重定向到外部 http 会被拒绝，无需改动。
- **实施决策（计划未明确）**：`http.request` / `websocket.open` 的准入权限从仅 `network` 放宽为 `network` 或 `localNetwork` 任一（新增 `require_network_permission`）。否则纯本机插件必须同时声明 `network`，而 `network` 权限强制 `allowedHosts` 非空，会逼出无意义的占位主机。
- **UI**：`PluginManagerPanel.tsx` 权限列表由原始 id 改为中文文案映射（network→网络、localNetwork→本机网络、browserSession→浏览器会话、cookies→Cookie）。
- **测试**：plugin_runtime.rs 新增 `local_network_permission_allows_loopback_plaintext_only`（回环 ws/http/[::1]/https 放行；192.168.1.5、0.0.0.0、`my.localhost.evil`、非白名单 https、ftp 拒绝）并在原白名单用例中补无权限时回环明文拒绝；plugin.rs 新增 v3 声明拒绝 / v4 声明可加载用例。
- **验证**：`cargo test` 187 通过；`npx tsc --noEmit` 通过。未执行真实本机 WebSocket 回显服务联调（无现成本机推理服务；规则层已被单元测试完整覆盖，实机联调留待 3.2 接入 sherpa-onnx/vLLM 场景时自然验证）。
