# 2.1 OCR 能力接入供应商框架

## 任务状态

- 状态：已完成
- 所属阶段：第二阶段-供应商能力框架升级
- 前置任务：无
- 关联任务：2.3（设置页消费 ocr 分类）、4.1 / 4.2（OCR 供应商的实际实现与消费）
- 是否可以并行：可与 1.1、2.2 并行

## 任务目标

"ocr" 成为供应商框架的正式能力类型：`ProviderDefaults` 有默认 OCR 供应商字段，插件 manifest 可声明 `ocr` 能力，`capabilities.rs` 有 `OcrProvider` 能力工厂（含内置 Windows 系统 OCR 与插件变体），插件运行时定义 `recognizeImage` 调用约定。本任务只建框架与内置系统 OCR 供应商，不改场景感知消费方（4.2 负责）、不做 PP-OCR 供应商（4.1 负责）。

## 当前情况

- 能力类型扩展机制已预留：`docs/rules/新增供应商与模型操作手册.md` 场景四描述了标准步骤（`ProviderDefaults` 加字段 + `default_provider_id`/`set_default_provider` 加分支 + profile 标记能力 + 前端 `ProviderCapability` 加字面量）。`llm` 能力就是按此接入的现成参照（`src-tauri/src/providers/mod.rs`）。
- `capabilities.rs` 现有 `FileRecognitionProvider` / `TranslationProvider` / `CustomizationProvider` 三个枚举 facade，均含 `Plugin { spec, profile }` 变体，插件经 `plugin_runtime::invoke` 调用命名操作（如 `translate`、`transcribeFile`）。
- 插件能力白名单：`plugin.rs::validate_manifest` 只认 `asr | translation | customization`；`PLUGIN_API_VERSION = 3` 精确匹配（决策：升 v4，兼容 v3，见重要记录 004）。
- OCR 数据结构已存在：`active_app_context/model.rs` 的 `OcrTextBlock` / `NormalizedRegion`（归一化坐标）；系统 OCR 实现在 `active_app_context/windows_ocr.rs`。
- 图像传输量评估：场景感知截图上限 960px PNG，Base64 后远低于 QuickJS 的 16MB 响应上限，经现有 `host.call` 字符串通道传输可行。

## 实施方案

1. **版本与白名单**：`PLUGIN_API_VERSION` → 4；`validate_manifest` 接受 `api_version ∈ {3, 4}`；能力白名单加入 `ocr`（仅 v4 允许声明，v3 声明 ocr 报错）。
2. **默认供应商**：`ProviderDefaults` 加 `ocr: String`（`#[serde(default)]`，默认指向内置系统 OCR profile id）；`default_provider_id` / `set_default_provider` / `normalize_settings` 加 `ocr` 分支。
3. **内置系统 OCR profile**：新增 `windows_ocr_profile()`（建议 id `system-ocr`，kind `builtin-windows-ocr`，capabilities `["ocr"]`，无配置项），加入 `builtin_profiles()`。
4. **能力工厂**：`capabilities.rs` 新增 `OcrProvider` 枚举：`System`（转发 `windows_ocr`）与 `Plugin { spec, profile }`；对外统一方法 `recognize(image_png: &[u8], purpose: &str) -> Result<Vec<OcrTextBlock>, String>`（通用接口，见重要记录 001）。注意依赖方向：`providers` 不应反向依赖 `active_app_context`，`OcrTextBlock`/`NormalizedRegion` 若被两侧共用，执行时评估是否上提到共享位置（如 `providers` 内定义、`active_app_context` 复用，或独立小模块），把结论写入执行记录。
5. **插件调用约定**：v4 插件的 `recognizeImage` 操作，payload `{ imageBase64, purpose }`，返回 `{ blocks: [{ text, region: { x, y, width, height } }] }`（坐标 0~1 归一化）；写入插件运行时的约定校验与错误消息。
6. **前端类型**：`ui/src/store/useProviderStore.ts` 的 `ProviderCapability` 加 `"ocr"` 字面量；`get_model_catalog` / 供应商列表输出自然携带（数据驱动，无需额外接线）。

## 执行步骤

3. 改 `capabilities.rs`：`OcrProvider` 枚举 + `ocr_for_with_plugin` 工厂 + 单元测试（无能力供应商返回结构化错误）。

## 执行记录

### 2026-07-18 完成实施

- **OCR 数据结构上提（关键结构决策）**：新增顶层模块 `src-tauri/src/ocr/`——`mod.rs` 定义 `NormalizedRegion`、`OcrTextBlock` 与 `normalize_text`（空白折叠），`windows.rs` 为原 `active_app_context/windows_ocr.rs` 整体迁移（git mv）。`active_app_context/model.rs` 与 `normalize.rs` 通过 `pub(crate) use crate::ocr::...` 再导出，原有引用路径全部不变；`providers` 依赖 `crate::ocr`，不反向依赖 `active_app_context`。
- **版本策略**：`PLUGIN_API_VERSION = 4`，新增 `PLUGIN_MIN_API_VERSION = 3`，校验接受 3..=4；`ocr` 能力仅 v4 允许声明（v3 声明报“ocr 能力需要插件 API 版本 4”）。
- **实施偏差（计划外但必要）**：原校验强制“插件至少声明一个模型”，而 OCR 按供应商而非模型选择，纯 OCR 插件没有模型条目。放宽为：能力仅含 `ocr`（可附带 `customization`）时允许 `models` 为空；声明 asr/translation 的插件仍必须带模型。
- **默认供应商与内置 profile**：`ProviderDefaults.ocr`（`#[serde(default)]`）；新增 `windows_ocr_profile()`（id `system-ocr`、kind `builtin-windows-ocr`、capabilities `["ocr"]`、无配置项）加入 `builtin_profiles()`；`default_provider_id` / `set_default_provider` / `normalize_settings` 均加 `ocr` 分支，缺省自动落到 `system-ocr`。
- **能力工厂**：`capabilities.rs` 新增 `OcrProvider`（`System` / `Plugin`）与 `ocr_for_with_plugin`；`recognize(image_png, purpose)` 统一接口。System 分支经 `spawn_blocking` 解码 PNG 后转发 `crate::ocr::windows::recognize`（非 Windows 返回结构化错误）；插件分支调用 `recognizeImage`，超时 60 秒。
- **插件调用约定**：payload `{ imageBase64, purpose }`；返回 `{ blocks: [{ text, region: { x, y, width, height }, confidence? }] }`，坐标 0~1；宿主侧解析做空文本过滤、坐标 clamp、confidence 缺省 1.0，格式错误消息指明期望字段。
- **前端**：`ProviderCapability` 加 `"ocr"`；`ProviderDefaults` 接口与 `FALLBACK_DEFAULTS` 加 `ocr` 字段。
- **测试**：plugin.rs（v4 纯 OCR 插件无模型可加载、v3 声明 ocr 拒绝、apiVersion 5 拒绝）；mod.rs（旧 JSON 归一化后 ocr 默认落 system-ocr、set_default_provider("ocr") 切换与拒绝）；capabilities.rs（工厂仅认内置 profile、blocks 解析归一化/clamp/错误消息、QuickJS 假插件 recognizeImage 端到端往返）。
- **验证**：`cargo test` 185 通过；`npx tsc --noEmit` 通过；`npm run ui:build` 通过。
