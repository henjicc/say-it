# 3.1 模型包格式与安装管理

## 任务状态

- 前置任务：1.1（models 目录落在数据根目录）
- 关联任务：3.2 / 3.3 / 4.1（模型包的消费方）、5.1（双击导入需覆盖模型包）
- 是否可以并行：与 2.3 可并行

## 任务目标

`.sayit` 支持第二种运行时形态 `runtime.kind = "model-pack"`：包内只有模型文件与 manifest（无 JS、无可执行文件），安装后生成对应的供应商 profile 与模型条目；同一 manifest 同时支持内嵌模型文件与"清单 + 宿主下载"两种形态（重要记录 002）。本任务完成格式定义、安装/卸载/校验/下载，不含任何推理引擎（3.2、4.1 负责）。

## 当前情况

- `.sayit` 现有格式：zip 包，`manifest.json`（`plugin.rs::PluginManifest`，`runtime.kind` 默认 `"javascript"`、`entrypoint` 必填）+ integrity（逐文件 SHA256）+ 可选 Ed25519 签名；`plugin_package.rs::install_from_path` 解压到 `plugins_dir()`，校验含**禁止原生可执行文件与动态库**（`.sayit 不能包含原生可执行文件或动态库`，约 488 行）——该校验对模型权重（.onnx/.mnn 等数据文件）不构成障碍，保持不放松。
- manifest 已支持 `models: Vec<ModelInfo>`（与 `shared/asr-models.json` 同构，含 `protocol`、`scenes` 等字段），插件模型经 `load_registry_from_with_trust` 合入模型目录并查重——模型包声明模型可直接复用这条通路。
- 安装入口：`plugin_management.rs::install_provider_plugin`（同步命令）；前端 `PluginManagerPanel.tsx`（文件对话框 + 未签名/信任确认）。
- 下载基建：`reqwest`（已含 `stream` feature）可做流式下载；数据目录 `models/` 子目录由 1.1 约定。

## 实施方案

### manifest 扩展（v4）

`runtime.kind = "model-pack"` 时：`entrypoint` 不要求；新增 `modelPack` 段：

```json
{
  "runtime": { "kind": "model-pack" },
  "modelPack": {
    "engine": "sherpa-onnx-online | sherpa-onnx-offline | ppocr-mnn",
    "files": [
      { "path": "encoder.int8.onnx", "sha256": "...", "sizeBytes": 123456,
        "download": { "url": "https://..." } }
    ],
    "params": { }
  }
}
```

- `engine` 是宿主内置引擎标识，宿主不认识则安装时报"需要升级软件"的结构化错误。
- 文件条目在包内存在 → 解压即用（仍按 integrity 校验）；不在包内且有 `download` → 安装后由宿主下载到该插件的模型目录，SHA256 校验一致后才可用。两种形态可混合。
- `params` 为引擎参数透传（如采样率、解码参数），由消费方（3.2/4.1）解释。
- 校验规则：model-pack 不得声明 `permissions` / `browserSession` / `network`；capabilities 只允许 `asr` / `ocr`。

### 安装与下载

1. 安装流程复用 `install_from_path`（解压、integrity、签名、信任），按 `runtime.kind` 分流校验逻辑。
2. 大包考虑：安装改为可发进度事件的异步命令（现有同步命令保留兼容或统一迁移，执行时评估改动面后二选一并记录）。
3. 下载器：流式写临时文件 + SHA256 边下边算 + 断点续传（HTTP Range）+ 进度事件；下载状态（未下载/下载中/已就绪/校验失败）随插件快照输出给前端。
4. 模型文件存放：`数据根/models/<插件id>/`（与 plugins 目录分开，卸载插件时一并清理，需用户确认——模型文件可能数百 MB，误删代价高）。
5. `PluginManagerPanel.tsx`：模型包条目显示类型徽标、总大小、下载/校验状态与进度、"下载模型"按钮（清单包）。

### 官方包构建

- 新增构建脚本（建议 `scripts/build-model-pack.mjs` 或 Rust xtask，执行时定）：输入模型文件与包描述 → 产出内嵌形态与清单形态两个 `.sayit`（同 id 同版本，文件条目差异仅在是否内嵌）+ integrity + 签名。首个真实包在 3.2（Paraformer）产出。

## 执行步骤

1. `plugin.rs`：manifest 结构加 `modelPack` 段与 model-pack 分支校验（含负例测试：带 entrypoint、带 network、声明 translation 的 model-pack 均拒绝）。
2. `plugin_package.rs`：安装分流；模型文件目录约定；卸载清理。
3. 新增下载器模块（建议 `providers/model_download.rs`）：下载、续传、校验、进度事件、状态持久化；单元测试用本地文件 URL 或 mock。
4. 插件快照（`PluginRegistrySnapshot`）扩展模型包状态字段；前端 `PluginManagerPanel.tsx` 展示与操作。
5. 构建脚本：产出双形态包；用一个小体积假模型包走通"安装内嵌包"与"安装清单包 + 下载"两条路径。
6. 验证：`cargo test`、`npx tsc --noEmit`、`npm run ui:build`；`tauri:dev` 下实际安装/卸载假模型包。
