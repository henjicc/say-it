# 4.1 PP-OCR 模型包化

## 任务状态

- 状态：已完成（安装与上下文调试交互待用户人工回归）
- 所属阶段：第四阶段-OCR供应商化
- 前置任务：2.1（ocr 能力类型）、3.1（模型包机制）
- 关联任务：4.2（消费方）、1.1（模型落数据目录）
- 是否可以并行：与 3.2/3.3 可并行（模型包机制就绪后）

## 任务目标

PP-OCR 从"安装目录 resources 内置"改为模型包分发：`ppocr-mnn` 引擎标识 + 官方 PP-OCRv6 tiny 模型包；安装该包后生成一个具有 `ocr` 能力的供应商 profile，模型文件位于数据目录。`ocr-rs`/MNN 推理引擎保持编译内置不变。

## 当前情况

- `src-tauri/src/active_app_context/ocr.rs`：det/rec/charset 三个文件（共约十几 MB）+ SHA256 常量写死；`model_root()` 默认取 `CARGO_MANIFEST_DIR/resources/ocr`，运行时由 `active_app_context/mod.rs:327` 的 `configure_model_root`（`OnceLock`，一次性）注入打包资源路径；引擎按任务构建即释放，worker 线程常驻。
- 打包：`tauri.conf.json` 的 `bundle.resources` 含 `"resources/ocr/*"`。
- OCR 引擎选择当前是 `OcrEngineKind { System, PpOcr }`，持久化在听写设置（4.2 负责迁到供应商框架；本任务保持该枚举工作，只改模型来源）。

## 实施方案

1. **模型包**：用 3.1 脚本把三个模型文件打成 `ppocr-mnn` 引擎的官方包（内嵌形态为主——文件小，无需清单形态也可同步产出）；manifest `capabilities: ["ocr"]`，`modelPack.files` 的 SHA256 取代代码里的常量。
2. **模型定位改造**：`ocr.rs` 的 `model_root()`/SHA256 常量改为从已安装模型包解析（经 3.1 的模型包查询接口：按引擎标识 `ppocr-mnn` 找到已就绪的包 → 文件路径 + manifest 内 SHA256）。`configure_model_root` 的 `OnceLock` 注入机制随之废弃或改为动态查询（插件可热安装，路径不能再一次性固化）。
3. **供应商 profile**：模型包安装后经现有 `merge_provider_profiles` 通路生成 `ocr` 能力供应商（显示名如「PP-OCR 本地识别」），出现在设置页 OCR 分区（2.3 已就绪则自动呈现）。
4. **过渡策略（已确定）**：安装器不再捆绑 `resources/ocr`，PP-OCR 与 Paraformer/SenseVoice 一样通过插件管理手动安装官方 `.sayit` 模型包。旧版选择为 PP-OCR 但未安装模型包时回退到系统 OCR；已安装包时自动迁移到模型下拉项。
5. 未安装 PP-OCR 包时：`OcrEngineKind::PpOcr` 分支返回结构化错误引导安装，系统 OCR 不受影响。

## 执行步骤

1. 制作官方 PP-OCR 模型包并在 3.1 机制下安装验证。
2. 改造 `ocr.rs` 模型定位（含包缺失/校验失败错误路径测试）；清理 `configure_model_root` 注入链路。
3. 接通供应商 profile 生成，确认设置页 OCR 分区出现该供应商。
4. 与用户确认过渡策略后实施（预置包或移除捆绑 + 引导安装）；同步修改 `tauri.conf.json` 的 resources 清单。
5. 验证：`cargo test`、上下文调试窗口（`desktop/context_debug.rs` 已有调试入口）实际跑一次 PP-OCR 识别；卸载模型包后确认回退错误与系统 OCR 可用。
