# 4.2 场景感知 OCR 接入供应商选择

## 任务状态

- 状态：待开始
- 所属阶段：第四阶段-OCR供应商化
- 前置任务：2.1（OcrProvider 工厂）、4.1（PP-OCR 供应商化）
- 关联任务：2.3（OCR 分区的默认供应商下拉即本任务的选择入口）
- 是否可以并行：否

## 任务目标

场景感知的 OCR 引擎选择从听写设置的 `OcrEngineKind` 枚举迁移到供应商框架：按 `defaults.ocr` 路由到系统 OCR / PP-OCR 模型包 / 第三方插件 OCR；选用插件（云端）OCR 时有一次性隐私确认（屏幕截图将发送到第三方服务）。旧设置值平滑迁移。

## 当前情况

- 引擎选择：`active_app_context/model.rs::OcrEngineKind { System, PpOcr }`，序列化字符串 `"system"` / `"ppocr"`；持久化在 `application/dictation.rs` 的 `active_app_context_ocr_engine`（约 99 行，默认值 System），消费点在捕获链路 `active_app_context/mod.rs::CaptureOptions::for_method(method, ocr_engine)` 与上下文调试窗口 `debug.rs`（支持临时覆盖引擎）。
- OCR 执行在专用 worker 线程（`ocr.rs` 的 `OCR_WORKER`），同步阻塞式管线，带 deadline 与取消轮询；插件 OCR 是异步网络调用，接入时要在 worker/调用侧做同步桥接（现有 `plugin_runtime` 的调用在 `capabilities.rs` 均为 async，场景感知链路本身由 `dictation.rs` 异步编排，执行时确认最小改动的桥接点并记录）。
- 2.1 已提供 `OcrProvider` 工厂（System / Plugin 变体），4.1 提供 PP-OCR 供应商；本任务把三者接到捕获链路。
- 场景感知截图约 ≤960px PNG，Base64 后体积远小于插件响应上限。

## 实施方案

1. **路由改造**：捕获链路的 OCR 分支改为按 `defaults.ocr` 供应商解析——内置系统 OCR profile → 现有 `windows_ocr` 路径；PP-OCR 供应商 → 现有 `ocr.rs` 管线；插件供应商 → `OcrProvider::Plugin.recognize(image, "active-app-context")`。`OcrEngineKind` 收缩为内部实现细节或废弃（执行时视 `debug.rs` 覆盖功能的改动面决定，倾向：调试窗口的引擎覆盖改为供应商覆盖）。
2. **设置迁移**：旧 `active_app_context_ocr_engine` 值在加载时映射为 `defaults.ocr`（`"system"` → 系统 OCR profile id，`"ppocr"` → PP-OCR 供应商 id；PP-OCR 未安装则回退系统 OCR），旧字段保留读取兼容一个版本。
3. **隐私确认**：`defaults.ocr` 切到插件供应商时（2.3 的下拉处触发），弹出确认弹窗说明"场景感知截图将发送至第三方服务"；确认状态按供应商 id 持久化，未确认不生效。插件 OCR 运行失败时记录日志并本次任务降级系统 OCR，不静默丢弃错误（错误可在上下文调试窗口看到）。
4. **超时**：插件 OCR 受场景感知现有 deadline 约束，超时按现有取消语义中止。

## 执行步骤

1. 捕获链路按供应商路由重构（保持系统/PP-OCR 行为逐字节等价，仅换选择机制）；`cargo test` 与上下文调试窗口回归两条内置路径。
2. 旧设置值迁移逻辑 + 测试（三种旧值 × 包安装与否）。
3. 隐私确认 UI 与持久化；插件 OCR 失败降级与日志。
4. 上下文调试窗口适配供应商覆盖。
5. 验证：`cargo test`、`npx tsc --noEmit`、`npm run ui:build`；调试窗口分别用系统 OCR / PP-OCR / 测试插件（可用假插件返回固定 blocks）各跑一次识别。
6. 用户实测：真实听写场景下场景感知润色正常。
