# 重要记录

## 记录 001：OCR 供应商接口按通用设计，本次只消费于场景感知

- 日期：2026-07-18
- 状态：已确定
- 背景：当前 OCR 只用于场景感知（截取活动窗口识别文字辅助润色）。接入第三方 OCR API 插件时，接口既可以按场景感知的最小需求设计，也可以按未来独立图片转文字功能通用化。
- 最终决定：接口按通用设计——输入图像（PNG 字节）+ 用途标识（purpose），输出文本块列表（文本 + 归一化区域坐标），与现有 `OcrTextBlock` / `NormalizedRegion`（`active_app_context/model.rs`）对齐；本次不做独立 OCR 功能 UI。
- 选择原因：用户明确要求预留独立 OCR 功能扩展空间；通用接口的额外成本很小（坐标结构已存在）。
- 影响任务：2.1、4.2

## 记录 002：模型包以内嵌模型为主，同步支持清单下载包

- 日期：2026-07-18
- 状态：已确定
- 背景：本地模型数百 MB，分发方式决定模型包 manifest 格式与安装流程。
- 最终决定：两种形态都支持。主形态为模型文件直接内嵌在 `.sayit` 包内（离线一个文件装完）；同一 manifest 结构同时支持文件条目只带下载描述符（URL + SHA256 + 大小）的清单包，安装后由宿主统一下载到数据目录。官方模型包构建时同步产出两种形态。
- 选择原因：用户选择"两种都支持，默认以模型包（内嵌）为主，同步生成清单包"。
- 影响任务：3.1、3.2、3.3、4.1、5.1

## 记录 003：数据目录全量迁移，AppData 只留指针文件

- 日期：2026-07-18
- 状态：已确定
- 背景：当前设置文件与插件在 `AppData\Local\com.henjicc.sayit`，OCR 模型在安装目录 resources，数据分散且不可自定义位置。
- 最终决定：所有数据（设置、插件、模型及后续一切内容）统一放在单一数据根目录内；用户自定义位置时整目录迁移（复制 → 校验 → 删旧）；AppData 原位置只保留一个指向新目录的指针文件用于启动引导与旧版兼容。迁移完成后软件自动刷新持有路径的模块，无法热刷新则提示用户或自动重启。
- 选择原因：用户明确要求"所有数据统一存放在一个文件夹内，移动数据只需迁移整个文件夹"。
- 影响任务：1.1、3.1、4.1

## 记录 004：插件 API 版本升到 v4，宿主同时接受 v3

- 日期：2026-07-18
- 状态：已确定
- 背景：`plugin.rs::validate_manifest` 当前对 `PLUGIN_API_VERSION = 3` 精确匹配。本次新增 ocr 能力、model-pack 运行时形态、localNetwork 权限，属于 manifest 语义扩展。
- 最终决定：`PLUGIN_API_VERSION` 升到 4，校验放宽为接受 3 或 4；v3 插件（只用旧能力）行为完全不变。model-pack 与 ocr/localNetwork 声明仅在 v4 下合法。
- 选择原因：已有发布的 v3 在线插件不能因宿主升级失效；精确匹配策略在多版本共存下不可持续。
- 影响任务：2.1、2.2、3.1、5.2

## 记录 005：Windows 下 sherpa-onnx 改用共享运行库

- 日期：2026-07-18
- 状态：已确定
- 背景：3.2 链接 PoC 中，`sherpa-onnx` 默认静态库本身可以构建并完成真实 Paraformer/SenseVoice 识别，但与现有 `ocr-rs + mnn-static` 同进程时，只要初始化 PP-OCR 就会稳定触发 `STATUS_ACCESS_VIOLATION (0xc0000005)`。这意味着原计划“两套推理运行时都静态链接”在当前 Windows 组合下不可发布。
- 最终决定：保留 PP-OCR/MNN 静态链接，sherpa-onnx 改用官方 shared 特性；4 个官方运行库 DLL 由构建脚本复制到 Cargo 测试目录，并通过 Tauri resources + NSIS 安装钩子落到安装目录根部。模型包仍然严禁携带 DLL/可执行文件，运行库只属于宿主安装包。
- 选择原因：共享 sherpa 运行库完整通过 PP-OCR 原生夹具、192 项 Rust 测试、真实 Paraformer/SenseVoice 识别与 NSIS 构建；默认静态方案存在确定性原生崩溃，不能用“测试单独通过”掩盖。
- 影响任务：3.2、3.3、发布构建

## 记录 006：供应商只负责配置，运行场景统一选择模型

- 日期：2026-07-18
- 状态：已确定
- 背景：供应商设置页曾为 ASR / OCR / 翻译分别提供“默认供应商”，但听写、字幕、转写和场景感知本身都有明确的模型选择入口；两套选择会产生冲突，也无法表达同一供应商的多个模型。
- 最终决定：移除 ASR / OCR / 翻译分区的默认供应商控件。供应商页面只负责安装、配置和启用；启用供应商的模型按 `scenes` 汇入对应下拉框，运行时由所选模型反查供应商。旧 `ProviderDefaults` 暂留作兼容与无显式模型参数的内部兜底，不再作为用户选择入口。
- 选择原因：模型才是实际运行单元；单一模型选择可以同时确定供应商、协议和能力，避免“默认供应商”和“已选模型”互相覆盖。
- 影响任务：2.3、4.2、5.2

## 记录 007：本地模型统一通过插件管理手动安装

- 日期：2026-07-18
- 状态：已确定
- 背景：Paraformer、SenseVoice 与 PP-OCR 都已经具备 `.sayit` 模型包描述，PP-OCR 旧版仍随安装器内置权重。继续预置会形成两种模型来源，也会让未选择 PP-OCR 的用户被动安装权重。
- 最终决定：安装器不再捆绑 PP-OCR 权重；三类本地模型都从“插件管理 → 安装 .sayit 包”安装，安装并启用后自动进入对应场景模型下拉。正式 Release 模型包由发布者私钥签名并作为独立资产分发。
- 选择原因：模型生命周期、数据目录、完整性校验、卸载和按需分发全部统一；应用安装包更小，用户只安装实际需要的本地模型。
- 影响任务：3.2、3.3、4.1、5.1、5.2
