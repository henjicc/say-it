# 实施方案：插件体系升级与本地模型扩展

本文件记录相对稳定的整体方案。任务状态与进度只看 `00-任务总览.md`。

## 一、现状（2026-07-18 调研，代码为准）

### 1. 插件体系（.sayit，API v3）

- 运行时：QuickJS（rquickjs 0.12），内存上限 64MB、栈 1MB、响应体上限 16MB、单次调用默认超时 10 分钟。核心文件：
  - `src-tauri/src/providers/plugin.rs`：manifest 结构与校验（`PLUGIN_API_VERSION = 3`，精确匹配；能力白名单校验在 `validate_manifest`，当前要求声明 `asr | translation | customization` 之一；权限白名单 `ALLOWED_PERMISSIONS = ["network", "browserSession", "cookies"]`；`plugins_dir()` = `app_local_data_dir()/plugins`）。
  - `src-tauri/src/providers/plugin_runtime.rs`：宿主 API（`host.call` 分发：`crypto.*`、`time.*`、`storage.*`（KV）、`resource.readBytes/readText`（仅插件目录内）、`cancel.isCancelled`、`http.request`、`websocket.open/send/close`）。网络仅放行 `https`/`wss`（`parse_allowed_url`），主机名需匹配 manifest 的 `allowedHosts`。
  - `src-tauri/src/providers/plugin_package.rs`：`.sayit` 为 zip 包；安装校验含逐文件 SHA256（integrity）、Ed25519 签名（signature/trust）、**禁止原生可执行文件与动态库**；`install_from_path` 解压到 plugins 目录。
  - `src-tauri/src/providers/plugin_secrets.rs`：会话密文走系统 keyring（`com.sayit.provider-plugin-session`），不在文件里。
  - `src-tauri/src/commands/asr/plugin_session.rs`：插件实时 ASR 会话（音频经 channel 喂给 JS 运行时，事件回发 `emit_asr_stream_event`）。
  - `src-tauri/src/application/plugin_management.rs`：`install_provider_plugin` / `list_provider_plugins` / `reload_provider_plugins` 等 Tauri 命令。
  - 前端：`ui/src/views/PluginManagerPanel.tsx`（安装/卸载 UI，经文件选择对话框）。
- 插件 manifest 可声明 `models`（合入模型目录）、`configFields`、`actions`、`browserSession`。

### 2. 供应商框架

- `src-tauri/src/providers/mod.rs`：`ProviderProfile { id, kind, capabilities, config, ... }`；`ProviderDefaults` 当前含 `asr` / `llm` / `translation` 三个默认供应商字段；`builtin_profiles()` = `funasr_profile()`（阿里云百炼，capabilities: asr+llm+translation）+ `groq_llm_profile()`。`has_capability` / `default_provider_id` / `set_default_provider` 已数据驱动。
- `src-tauri/src/providers/capabilities.rs`：能力工厂枚举 facade——`FileRecognitionProvider` / `TranslationProvider` / `CustomizationProvider`，各含 `AlibabaCloud` 与 `Plugin` 两个变体。**插件翻译通道后端已可用**（`translate` invoke 操作），缺的是 UI 分类与展示。
- 模型注册表：仓库根 `shared/asr-models.json` → `src-tauri/src/providers/registry.rs`；模型带 `protocol` 字段，实时会话状态机（`commands/asr`）协议无关。目录经 `application/catalog.rs` 的 `get_model_catalog` 输出给前端，前端不自行推断能力。
- 扩展流程见 `docs/rules/新增供应商与模型操作手册.md`（场景一~四）。

### 3. OCR 现状

- 仅服务「场景感知」（截取活动窗口识别文字辅助润色）。`src-tauri/src/active_app_context/ocr.rs`：PP-OCRv6 tiny 经 `ocr-rs`（vendor 补丁版，MNN 静态链接，仅 Windows）；模型文件（det/rec/charset，共约十几 MB）打包在安装目录 `resources/ocr`，SHA256 常量写死在代码里；引擎按任务构建、任务结束即释放；独立 worker 线程。
- `windows_ocr.rs`：Windows 系统 OCR（WinRT `Media_Ocr`）。引擎选择 `OcrEngineKind { System, PpOcr }`（`active_app_context/model.rs`），持久化在听写设置 `dictation.rs` 的 `active_app_context_ocr_engine`，不走供应商框架。

### 4. 设置与数据

- 设置页：`ui/src/views/SettingsView.tsx` 的 `provider` 标签页 =「密钥与识别」→ `SettingsProviderPanel.tsx`，单一 `SettingsSection title="识别供应商"`，内部硬编码阿里云百炼面板 + 插件供应商列表。LLM 供应商在独立的 `SettingsLlmPanel.tsx`。
- 持久化：`src-tauri/src/persistence.rs`，`say-it-state.json` 在 `app_local_data_dir()`（`AppData\Local\com.henjicc.sayit`）；插件在同目录 `plugins/`；OCR 模型在安装目录 resources。**数据分散在三处**。
- 打包：`src-tauri/tauri.conf.json`，NSIS，无 `fileAssociations`；已装 `tauri-plugin-single-instance`。

### 5. 现存问题

1. 插件只能做在线 API 连接器，本地模型无扩展通道；OCR 能力完全游离于供应商框架之外。
2. 插件网络只放行 https/wss，无法连接本机 GPU 服务（如 vLLM 的 `ws://127.0.0.1`）。
3. 「识别供应商」单一分类无法容纳 ASR/OCR/翻译三类；翻译插件后端已支持但 UI 无处呈现。
4. 数据分散、不可自定义位置，模型大文件将来会挤占 C 盘。
5. `.sayit` 安装只能从软件内文件对话框导入。

## 二、总体方案

### 路线：内置推理引擎 + 模型包（已确认，不做代码型本地插件）

值得本地跑的 ASR 模型（Paraformer Online、SenseVoice-Small，以及未来的 Fun-ASR-Nano/Qwen3-ASR 离线版）全部收敛到 sherpa-onnx 运行时。因此本地模型扩展 = **一个内置 Rust 引擎（`sherpa-onnx` crate，静态链接，首版仅 CPU INT8）+ N 个模型包**。模型包复用 `.sayit` 的 zip/签名/SHA256 机制，新增 `runtime.kind = "model-pack"`：包内只有模型权重 + manifest，**无 JS 代码、无可执行文件**。OCR 同理：`ocr-rs` 引擎保持内置，PP-OCR 模型文件从安装目录 resources 改为模型包分发。

### 模型包分发（已确认决策，见重要记录 002）

- 主形态：模型文件直接内嵌在 `.sayit` 包中（离线可装、一个文件即全部）。
- 同步支持：清单包（manifest 只含文件的下载 URL + SHA256 + 大小，安装后由宿主下载到数据目录，统一实现断点续传与校验）。
- 同一 manifest 文件条目结构统一为 `{ path, sha256, sizeBytes, download?: { url } }`：条目在包内则直接解压，不在包内且有 download 则下载。官方模型包由构建脚本同时产出两种形态。

### 插件能力矩阵（升级后）

| 能力 | 在线 API 插件（JS） | 模型包插件 | 内置 |
|---|---|---|---|
| asr | 已支持（实时+文件） | 新增（sherpa-onnx 协议族） | 阿里云百炼 |
| translation | 后端已支持，本次补 UI | 不在本次范围 | 阿里云百炼 |
| ocr | 新增（`recognizeImage` 调用约定） | 新增（PP-OCR 包） | Windows 系统 OCR |
| customization | 已支持 | — | 阿里云百炼 |

OCR 插件接口按通用设计（输入图像 + 用途标识，输出文本块 + 归一化区域坐标），本次只消费于场景感知，为将来独立图片转文字功能预留（重要记录 001）。

### 数据目录（已确认决策，见重要记录 003）

- 单一数据根目录，默认 `AppData\Local\com.henjicc.sayit`，内含 `say-it-state.json`、`plugins/`、`models/`（新增）等全部数据。
- 用户可在设置中改到任意位置；迁移 = 整目录复制 → 校验 → 删除旧目录 → 写指针文件；AppData 原位置只留一个指针文件（如 `data-root.json`）供启动引导与旧版本兼容。
- 迁移完成后软件自动刷新各持有路径的模块；无法热刷新的部分提示或自动重启。

### 插件 API 版本策略

`PLUGIN_API_VERSION` 升到 4（新增 ocr 能力、model-pack 形态、localNetwork 权限），宿主校验放宽为**接受 3 或 4**；v3 在线插件行为完全不变。`adapt-sayit-provider` 技能同步升级。

## 三、阶段划分与关系

1. **第一阶段-统一数据目录**：最先做——后续模型包、PP-OCR 包都落在新目录结构里，避免返工二次迁移。
2. **第二阶段-供应商能力框架升级**（2.1 OCR 能力 / 2.2 回环权限 / 2.3 设置页三分类）：纯框架与 UI，2.1、2.2 互相独立且不依赖第一阶段，可与 1.1 并行；2.3 依赖 2.1（要有 ocr 能力类型才能分出 OCR 供应商栏目）。
3. **第三阶段-模型包与本地ASR**：3.1 模型包格式（依赖 1.1 的 models 目录）→ 3.2 sherpa-onnx 实时（依赖 3.1）→ 3.3 SenseVoice（依赖 3.2 的引擎接入）。
4. **第四阶段-OCR供应商化**：4.1 PP-OCR 模型包化（依赖 2.1 + 3.1）→ 4.2 场景感知接入供应商选择（依赖 4.1）。
5. **第五阶段-安装体验与文档收尾**：5.1 文件关联（依赖 3.1，导入确认 UI 需覆盖模型包）；5.2 文档与技能同步（最后做）。

## 四、兼容与风险控制

### 兼容要求

- v3 在线插件不失效；`say-it-state.json` 旧结构经 `#[serde(default)]` 平滑升级（已有先例：`persistence.rs` 的 legacy identifier 迁移）。
- 旧版本降级安装时：指针文件存在但旧版不认识 → 旧版会在 AppData 原位置找不到数据。迁移向导中明确告知此风险即可，不做双写。
- `dictation` 设置里现有的 `active_app_context_ocr_engine` 字段迁移到供应商框架后需保留读取兼容（旧值映射到新默认 OCR 供应商）。

### 主要风险

| 风险 | 应对 |
|---|---|
| sherpa-onnx crate 在 Windows/CI 的静态链接与产物体积（预计 +10~20MB） | 3.2 开始前先做最小 PoC（cargo check + 一次真实识别）；失败则评估 `SHERPA_ONNX_LIB_DIR` 自编译路径，结论写入重要记录 |
| 数据目录迁移中断/目标磁盘空间不足 | 先复制后删除；复制完成并校验前旧目录不动；任何一步失败回退指针不写、旧目录保留 |
| 大 `.sayit` 包（内嵌数百 MB 模型）的解压与校验耗时 | 安装流程异步化 + 进度事件；SHA256 校验流式计算 |
| 云端 OCR 隐私（截图外发） | 场景感知选用插件 OCR 供应商时一次性显式确认，确认状态持久化（4.2） |
| 设置页重构破坏现有配置流 | 2.3 只动布局分类，不动 store 与命令层；逐面板回归 |

### 回退原则

每个任务独立提交；涉及持久化结构的任务（1.1、4.2）在提交信息中注明 schema 影响。回退按提交粒度 revert，数据目录迁移功能回退时指针文件机制保持向后可读（找不到指针则用默认目录）。

## 五、全局开发约束

- 遵循仓库 `CLAUDE.md` 全部约定；新增/扩展模型先读 `docs/rules/新增供应商与模型操作手册.md`，完成后由 5.2 更新该手册。
- 验证命令基线：`cd src-tauri && cargo check && cargo test`、`npx tsc --noEmit`、`npm run ui:build`；涉及实时链路与 UI 的人工回归用 `npm run tauri:dev`（端口 5155），麦克风/快捷键/注入类验证交由用户实测。
- UI 必须复用 `ui/src/index.css` 令牌与 `components/ui` 基础组件；多入口注意 `ui/index.html` 与 `ui/indicator.html`。
- 不引入 GPU 依赖；不放松 `.sayit` 禁止原生二进制的校验。

## 六、最终验收条件

1. 数据目录可自定义并整体迁移，迁移后听写、字幕、插件、OCR 功能全部正常。
2. 设置页呈现「ASR 供应商 / OCR 供应商 / 翻译供应商」三分类，仅负责配置与启用；各运行场景从聚合后的模型下拉选择实际模型与供应商。
3. 安装官方 Paraformer Online 模型包后，断网状态下实时听写可用；SenseVoice 包支持整句/文件识别。
4. PP-OCR 以模型包形式安装后场景感知正常；未安装时系统 OCR 兜底。
5. v3 在线插件（现有已发布插件）安装与运行不受影响；新 v4 插件可声明 ocr/translation 并出现在对应分类。
6. 双击 `.sayit` 文件可唤起软件并进入导入确认流程。
7. `cargo test`、`npx tsc --noEmit`、`npm run ui:build` 全绿；操作手册与 `adapt-sayit-provider` 技能已同步。
