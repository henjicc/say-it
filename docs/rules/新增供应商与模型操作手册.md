# 新增供应商与模型操作手册

本手册描述「说吧！」在完成《供应商与模型扩展性改造》后，新增识别模型/协议族/供应商/能力类型时的确切步骤。步骤以任务 4.1 用 `paraformer-realtime-v2` 真实走通一次流程的实际操作为准，不是设计推测。

## 适用范围与边界

- 覆盖：实时识别（听写实时 / 字幕）与文件识别（听写文件 / 录音识别）的模型新增、实时协议族新增、供应商新增、能力类型新增。
- 文件识别、翻译和热词 customization 的业务边界统一在 `providers/capabilities.rs`；供应商协议实现仍留在各自 provider 模块内。
- 当前只有一个真实供应商，能力工厂采用小型枚举 facade，不是动态插件系统；新增供应商需要扩展工厂，但不应修改听写、字幕、转写或对比状态机。

## 场景一：新增同协议族的新模型

**适用**：模型使用已在 `shared/asr-models.json` 中出现过的 `protocol` 值（如 `dashscope-duplex`、`file-async-oss` 等），只是模型名/版本不同。

**步骤**：

1. 在仓库根 `shared/asr-models.json` 追加一条记录，字段含义：
   - `id`：模型名（与云端调用时的 `model` 参数完全一致）。
   - `label`：下拉展示名。
   - `providerId`：供应商标识（当前固定为 `"funasr"`，对应 `FUNASR_PROVIDER_ID`；新供应商见场景三）。
   - `category`：`"realtime"` 或 `"file"`。
   - `protocol`：已存在的协议族字符串之一。
   - `supportsVocabulary` / `supportsAlignmentTimestamps`：按官方文档如实填写模型的真实能力。
   - `scenes`：模型出现在哪些下拉（`dictationRealtime` / `subtitles` / `dictationFile` / `transcription`），`subtitles` 场景由 `SUBTITLE_ASR_MODEL_OPTIONS` 消费，出现在实时字幕的模型下拉。
   - `isDefaultRealtime` / `isDefaultFile`：是否作为该类别的默认模型，全表只能有一个 `true`。
2. 若模型 `supportsVocabulary: true` **且**需要为其单独维护云端热词表：在 `src-tauri/src/providers/alibabacloud/customization.rs` 的 `VOCABULARY_TARGETS` 追加一行 `(模型名, 前缀)`，前缀不与其他行重复；追加后需在设置页重新保存一次热词，才会为新模型生成独立词表。
   - 注意 `supportsVocabulary: true` 不强制要求这一步：`VOCABULARY_TARGETS` 之外的模型仍会正常识别，只是不会下发 `vocabulary_id` 参数（这是 Paraformer 系列的既有约定，见该文件注释）。
3. 若仓库内测试对模型总数写了断言（如 `src-tauri/src/providers/registry.rs` 的 `test_model_count`），同步改成新的数量。这不是额外的架构接线，只是数据变化导致的测试快照更新。
4. 跑验证命令（见下文"通用验证清单"）。

**涉及文件（真实操作记录）**：`shared/asr-models.json`（新增一条）+ `src-tauri/src/providers/registry.rs`（`test_model_count` 断言值 +1）。未触碰任何协议/连接器/命令层代码。

## 场景二：新增同供应商的新协议族

**适用**：模型使用了该供应商尚未支持过的 WebSocket 协议（例如阿里云未来推出第三种实时协议）。

**步骤**：

1. 在 `shared/asr-models.json` 给新模型填一个新的 `protocol` 字符串。
2. `src-tauri/src/providers/alibabacloud/protocol.rs` 的 `RealtimeAsrFamily` 枚举新增一个变体。
3. `src-tauri/src/providers/registry.rs` 里所有按 `protocol` 字符串 match 的函数（`realtime_asr_family` 等）追加对应分支。
4. `src-tauri/src/providers/alibabacloud/realtime.rs` 新增一个实现 `RealtimeAsrConnector` trait 的结构体（`connect_request` / `start_messages` / `audio_message` / `finish_message` / `parse_message`），工厂函数 `realtime_connector()` 追加一个 match 分支。
5. 若新协议的热词/静音测试等派生判断规则不同，检查 `registry.rs` 里其余查表函数（`supports_vocabulary`、`realtime_family_supports_silence_test`）是否也需要新分支。
6. `commands/asr.rs` 的会话循环本身不需要改（已是协议无关的，只消费 `AsrEvent`）。

**涉及文件**：注册表 1 行 + `protocol.rs`（枚举）+ `registry.rs`（match 分支）+ `realtime.rs`（新连接器 + 工厂分支）。

## 场景三：新增新供应商

**适用**：接入阿里云百炼之外的第二家识别服务。

**步骤**：

1. 新建 `src-tauri/src/providers/<供应商>/` 模块，参考 `providers/alibabacloud/` 的组织方式实现该供应商的连接器（若协议与阿里云完全不同，新增一个新的 `RealtimeAsrConnector` 实现）。
2. `src-tauri/src/providers/mod.rs`：
   - 新增一个 `<供应商>_profile()` 内置 profile 构造函数（参考 `funasr_profile()`），加入 `builtin_profiles()` 清单，使其能被 `normalize_settings` 自动补齐且不被删除。
   - `realtime_connector_for()` 的 `match kind` 追加该供应商的 `kind` 分支。
3. `shared/asr-models.json` 追加该供应商的模型，`providerId` 填新供应商标识。
4. 若该供应商有专属配置项（不同的鉴权方式、专属参数）：前端新增一个设置面板组件（参考 `ui/src/views/SettingsProviderPanel.tsx` 与 `FunAsrHotwordsPanel.tsx` 的结构），当前 `SettingsProviderPanel` 是单供应商 UI（`provider?.id === "funasr"` 的热词面板即硬编码在其中），多供应商 UI 需要按 profile 列表渲染多个面板，这部分尚未泛化，需要新写。
5. 按真实能力实现 `providers/capabilities.rs` 的对应枚举分支：文件识别实现同步或异步生命周期方法，翻译实现标准化流式结果，热词实现可选 customization 方法；不支持的能力保持工厂返回结构化“不支持”错误。
6. 前端不读取注册表，也不按 `protocol`/模型名推断能力；`get_model_catalog` 会把 Rust 校验后的模型、场景、能力与供应商描述输出给前端。
7. 热词页面调用 `provider_*_hotwords` 并携带 provider ID；不要新增供应商专属公共命令。`funasr_*` 兼容命令已在 5.1 删除。

**涉及文件**：新 provider 模块 + `providers/mod.rs`（内置 profile + 工厂分支）+ 注册表数据 + （按需）新设置面板 + （按需）新文件转写通路。

## 场景四：新增能力类型（如 LLM 后处理）

当前机制已预留、未实现任何后处理功能。`ProviderDefaults` 结构、`has_capability` / `default_provider_id` / `set_default_provider` 均已数据驱动，不写死 `"asr"`。真正接入时：

1. `src-tauri/src/providers/mod.rs`：
   - `ProviderDefaults` 新增字段（参考现有 `llm: String`，务必加 `#[serde(default)]` 保证旧 JSON 兼容）。
   - `default_provider_id` / `set_default_provider` 的 `match capability` 追加分支（`llm` 分支已存在，可直接复用）。
2. 新供应商 profile 的 `capabilities` 数组标记新能力字符串（如 `["llm"]`），使其能被 `has_capability` 识别。
3. 新增该能力对应的命令层（如识别结果后处理的 Tauri 命令），复用 `resolve_provider_id` + `find_profile` 取配置，不要绕过。
4. 前端 `ui/src/store/useProviderStore.ts` 的 `ProviderCapability` 类型加新的字面量（`"llm"` 已存在）；有效供应商由 Rust 目录的 `effectiveCapabilities` 输出，前端不得自行回退选择。
5. 若需要专属 UI（如后处理开关面板），新增独立组件，不要塞进现有 `SettingsProviderPanel`。

**涉及文件**：`providers/mod.rs`（`ProviderDefaults` 字段）+ 新供应商/命令层 + 前端 `ProviderCapability` 类型 + （按需）新 UI 面板。

## 通用验证清单

每次改动后按顺序跑：

```bash
cd src-tauri && cargo check && cargo test
npx tsc --noEmit
npm run ui:build
```

涉及实时协议/连接器/命令层改动时，额外用 `npm run tauri:dev`（端口固定 5155）做人工回归，覆盖：

- 新模型出现在预期下拉（听写实时 / 字幕 / 听写文件 / 录音识别，按 `scenes` 字段）。
- 一次听写注入（选中新模型，说一句话，确认文字被正确注入）。
- 一次实时字幕开启（选中新模型，确认字幕正常滚动）。
- 若涉及热词：设置页保存一次热词，确认新模型对应词表创建成功（如场景一步骤 2 适用）。
- 涉及麦克风、快捷键、文字注入的验证必须由用户实测，不要自行模拟。
