# 新增供应商与模型操作手册

本手册以当前 API v4、模型包与场景模型目录的真实实现为准。新增前先判断属于哪一类，避免把模型数据、在线连接器和宿主推理引擎混在一起。

## 当前边界

- 供应商页面只负责安装、配置和启用，不选择“默认供应商”。启用供应商的模型按 `scenes` 汇入对应下拉框，运行时由模型反查供应商。
- JavaScript 在线插件使用 `.sayit` 的 `provider-plugin` 形态；本地权重使用无代码的 `model-pack` 形态。模型包不得携带 EXE、DLL、SO、DYLIB、WASM 或脚本运行时。
- 插件与模型都安装到 `application/data_root.rs` 管理的数据根目录；插件在 `plugins/`，模型权重在 `models/<插件 ID>/`。
- 宿主接受 JavaScript 插件 API v3 和 v4。v3 仅保留旧 ASR/翻译/customization 兼容；`ocr`、`localNetwork`、`model-pack` 只允许 v4。

## 场景一：现有在线协议增加模型

### 内置供应商

1. 在 `shared/asr-models.json` 新增模型，复用已有 `protocol`，准确声明 `category`、`providerId`、`scenes` 与能力标志。
2. 若模型需要阿里云热词表，在 `src-tauri/src/providers/alibabacloud/customization.rs::VOCABULARY_TARGETS` 增加目标；不需要则不要登记。
3. 同步 `src-tauri/src/providers/registry.rs` 中模型数量等数据快照测试。
4. 不修改前端模型数组；`application/catalog.rs::get_model_catalog` 是前端目录来源。

### JavaScript 供应商插件

直接在插件 `manifest.json.models` 增加模型。类别、协议、场景必须采用以下组合：

| 能力 | category | protocol | 至少一个场景 |
|---|---|---|---|
| 实时 ASR | `realtime` | `plugin-realtime-v1` | `dictationRealtime` / `subtitles` |
| 文件 ASR | `file` | `plugin-file-v1` | `dictationFile` / `transcription` |
| 翻译 | `translation` | `plugin-translation-v1` | `subtitleTranslation` |
| OCR | `ocr` | `plugin-ocr-v1` | `activeAppContext` |

每个模型的 `providerId` 必须等于 `provider.id`。完成后按仓库技能 `.codex/skills/adapt-sayit-provider/SKILL.md` 重新校验、签名和归档。

## 场景二：在线供应商增加新协议族

只有内置供应商的协议需要改宿主：

1. `shared/asr-models.json` 使用新的 `protocol`。
2. 在对应供应商协议模块增加协议枚举和连接器；阿里云实时 ASR 对应 `src-tauri/src/providers/alibabacloud/protocol.rs` 与 `realtime.rs`。
3. 在 `src-tauri/src/providers/registry.rs` 的协议查表函数增加映射。
4. 连接器继续输出统一 `AsrEvent`；不得在 `commands/asr` 增加按模型名分支。

JavaScript 插件不能自造新的宿主协议字符串，应继续使用四种 `plugin-*-v1` 协议，在连接器内部适配上游差异。

## 场景三：新增在线供应商

优先使用 `adapt-sayit-provider` 生成 API v4 JavaScript 插件，不修改应用源码：

1. 按技能调研鉴权、网络端点、实时/文件/翻译/OCR 契约。
2. `provider.capabilities` 只声明真实能力；配置字段和密钥使用 `configFields`，敏感字段标记 `secret: true`。
3. 公网请求声明 `network` 与最小 `allowedHosts`。仅连接本机 `127.0.0.1`、`localhost`、`[::1]` 的 HTTP/WS 时声明 `localNetwork`；它不允许局域网地址。
4. OCR 实现 `invoke({ operation: "recognizeImage", payload })`，返回 0~1 归一化区域；翻译实现 `invoke({ operation: "translate", payload })`。
5. 插件安装并启用后，模型自动进入场景下拉；不要新增“默认供应商”配置。

只有必须复用宿主原生能力且 JavaScript 沙箱无法满足时，才新增 `src-tauri/src/providers/<供应商>/` 内置模块，并通过 `ProviderProfile`、`capabilities.rs` 和模型目录的公共边界接线。

## 场景四：新增能力类型；OCR 实例

OCR 已按这条路线落地，可作为新能力模板：

1. `src-tauri/src/providers/mod.rs` 的 `ProviderProfile.capabilities`、设置归一化与内置 profile 支持能力字符串；持久化新增字段必须带 `#[serde(default)]`。
2. `src-tauri/src/providers/capabilities.rs` 定义能力 facade。OCR 的 `OcrProvider` 把系统 OCR、PP-OCR 模型包和 JavaScript 插件收敛到统一接口。
3. `src-tauri/src/providers/plugin.rs::validate_manifest` 声明 API 版本、能力、模型类别/协议/场景的合法组合。
4. `src-tauri/src/providers/plugin_runtime.rs` 只提供通用宿主 API；供应商差异留在插件中，禁止按插件 ID 特判。
5. `ui/src/store/useProviderStore.ts` 扩展 `ProviderCapability`；设置页只展示配置与启用，运行场景用模型下拉。
6. OCR 插件输入 PNG Base64 和 `purpose`，返回 `{ blocks: [{ text, region, confidence? }] }`；场景感知发送截图前还受 `application/dictation.rs` 的供应商隐私授权门禁。

## 场景五：新增本地 sherpa-onnx 模型包

先判断宿主是否已有对应引擎：

- 流式 Paraformer：`engine = sherpa-onnx-online`，模型协议 `local-sherpa-online`。
- 整句/文件 SenseVoice：`engine = sherpa-onnx-offline`，模型协议 `local-sherpa-offline`。
- PP-OCR：`engine = ppocr-mnn`，模型协议 `local-ppocr-mnn`。

已有引擎只需新增 `model-packs/<名称>.json` 描述，不改前端。描述格式参考：

- `model-packs/paraformer-online-int8.json`
- `model-packs/sensevoice-small-int8.json`
- `model-packs/ppocr-v6-tiny.json`

关键约束：

1. `manifest.apiVersion = 4`、`runtime.kind = model-pack`，无 `entrypoint`、权限、供应商配置或操作。
2. 每个 `modelPack.files` 条目必须有相对 `path`、真实 `sha256`、`sizeBytes`；可下载包再加 HTTPS `download.url`。
3. `sources` 与 `modelPack.files` 一一对应，源路径相对描述文件解析。
4. online 参数至少包含 `encoder`、`decoder`、`tokens`；offline 至少包含 `model`、`tokens`、`vadModel`；PP-OCR 至少包含 `detModel`、`recModel`、`charset`。参数值必须指向已声明文件。
5. online 模型只能声明实时场景；offline 可分别声明实时与文件模型；PP-OCR 只能声明 `activeAppContext`。

若新模型无法由现有三种引擎加载，属于“新增宿主本地引擎”，必须同时扩展 `plugin.rs::validate_model_pack`、`providers/local_asr/mod.rs` 或 OCR 推理模块、模型目录路由、运行库打包和真实模型夹具；不能只发明一个 engine 字符串。

## 场景六：制作官方模型包

1. 准备模型文件并计算 SHA256 与字节数，填入描述文件。正式私钥不得进入仓库。
2. 在仓库根目录用构建器生成并签名：

```powershell
$cargoArgs = @(
  'run'
  '--manifest-path'
  'src-tauri/Cargo.toml'
  '--features'
  'model-pack-builder'
  '--bin'
  'build_model_pack'
  '--'
  'model-packs/paraformer-online-int8.json'
  'C:/tmp/sayit-model-pack-output'
  'official-publisher-key-id'
  'C:/secure/sayit-publisher.pem'
)
& cargo @cargoArgs
if ($LASTEXITCODE -ne 0) { throw "模型包构建失败：$LASTEXITCODE" }
```

3. 构建器总是生成 `*-embedded.sayit`。仅当每个文件都有 HTTPS 下载地址时，额外生成 `*-manifest.sayit`；PP-OCR 当前只有内嵌包。
4. 内嵌包包含权重，可离线安装；清单包只含清单，安装后在插件管理点击“下载模型”，宿主使用 Range 断点续传并复核大小与 SHA256。
5. 正式 Release 使用发布者私钥签名并作为独立资产分发。应用安装器不捆绑 Paraformer、SenseVoice 或 PP-OCR 权重。
6. 安装入口统一为“设置 → 插件管理 → 安装 .sayit 包”或双击 `.sayit`；安装并启用后才会进入模型下拉。

## 验证清单

按改动范围由低到高执行：

```powershell
Set-Location 'src-tauri'
cargo test
Set-Location '..'
npx tsc --noEmit
npm run ui:build
npm run tauri:build
```

- JavaScript 插件：运行技能的 `validate_plugin.py`、`smoke_test_plugin.py`，再签名、归档并由宿主安装预览校验。
- 模型包：构建器校验源文件；宿主测试覆盖 manifest、签名、完整性和引擎参数；涉及新引擎时必须跑真实模型音频/图像夹具。
- 人工回归：安装并启用后检查对应场景模型下拉；测试实际识别/翻译/OCR；断网验证本地模型。鼠标、麦克风、快捷键、文字注入与双击文件关联由用户实测。
