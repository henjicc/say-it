# 给模型注册表新增能力声明字段的完整改动清单

## 场景

要让某个模型能力（如 `supportsContext`：是否接受上下文增强）被宿主感知并按声明下发时，
新增一个 `ModelInfo` 字段。字段分散在共享数据、Rust、前端和插件契约四处，漏改任何一处
都不会编译报错，只会静默失效或让既有插件验签失败。

## 必改清单

1. `src-tauri/src/providers/registry.rs::ModelInfo`
   - 新字段必须是 `Option<T>` 且带 `#[serde(default, skip_serializing_if = "Option::is_none")]`。
   - 原因：插件签名载荷是对 `PluginManifest` **重新序列化**得到的（`plugin_package::signing_payload`）。
     缺省字段若被序列化成 `null`，此前所有已签名插件与模型包的载荷都会改变，用户端表现为
     「插件签名验证失败」，全部集体失效。这是有过前科的坑（提交 `1ae80b0`）。
   - 同时提供 `pub fn supports_xxx(model)`，明确写出「未声明时」的兜底语义。
2. `shared/asr-models.json`：只在为 `true` 的模型上写出该键，保持与 `skip_serializing_if` 的语义一致。
3. `src-tauri/src/application/catalog.rs`
   - `ModelCatalogItem` 增加已解析成 `bool` 的字段（前端不感知「未声明」状态）。
   - **`CATALOG_VERSION` 必须 +1**，目录结构变了。
4. `ui/src/features/asr/modelRegistry.ts::ModelInfo`：补上同名字段，否则前端类型与后端契约脱节。
5. `src-tauri/src/providers/plugin_package.rs` 的
   `signing_payload_ignores_absent_optional_model_fields` 测试：把新字段加进被检查的字段列表，
   用测试锁死第 1 条。
6. 文档：`docs/rules/新增供应商与模型操作手册.md` 与
   `.claude/skills/adapt-sayit-provider/references/plugin-api.md`（**`.codex/` 下有一份完全相同的副本，
   两份必须同步改**，可用 `diff -rq` 核对）。

## 运行时下发的位置

内置模型查 `registry::model_info`，插件与模型包查 `PluginRegistry::model`——两个注册表是分开的，
只查其中一个会让插件模型的声明永远读不到。需要同时覆盖时把查询收在应用层（例如
`application::customization::capabilities_of`），供应商层只负责下发拿到的内容，不重复判断能力。
