# Windows UIA 大型辅助功能树捕获超时优化

## 典型现象

- 记事本等简单应用可以捕获，Chrome、Edge、VS Code 等应用频繁在 800ms 截止时超时。
- 一次慢捕获后，连续切换到其他软件测试也容易继续超时。

## 根因

1. UI Automation 的属性、树导航和 TextPattern 都可能产生跨进程调用。逐节点分别读取 `IsPassword`、`IsOffscreen`、`Name` 等属性，会把 300 个节点放大为上千次调用。
2. 先遍历一次寻找 Document、再从根节点遍历正文，会重复扫描大型辅助功能树。
3. 异步超时只能停止等待结果，不能中断已经进入 COM/UIA 的同步调用。单工作线程仍会被慢调用占用，后续请求只能排队超时。
4. 到硬截止后直接把已经得到的应用名、窗口标题和部分正文全部标记为超时，会丢弃仍有价值的降级结果。

## 稳定做法

- 使用 `IUIAutomationCacheRequest` 批量缓存常用属性，树导航后只读取缓存值。
- 优先读取焦点、选区和焦点祖先；在同一次祖先上行中寻找 Document，避免重复遍历。
- 文档可见范围和树节点都设置上限，并在硬截止前预留返回结果的时间。
- 达到软截止时返回已捕获的部分内容并标记截断，不等到硬截止才停止。
- 每次捕获使用独立 MTA 工作线程，同时限制并发数；某个应用的 UIA 调用卡住时，不阻塞后续捕获。
- 密码属性必须包含在缓存请求中；缓存缺失或读取失败时按敏感节点处理，不能为了性能降低隐私保护。

## 验证重点

- 分别测试首次捕获、连续快速捕获和一次超时后的下一次捕获。
- 关注状态、耗时、访问节点数和截断标记，不只检查是否返回正文。
- 浏览器和 Electron 应用应能在正文不完整时至少返回应用名、窗口标题、焦点或附近文本。
