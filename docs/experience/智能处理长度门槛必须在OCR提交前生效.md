# 智能处理长度门槛必须在 OCR 提交前生效

当前软件上下文使用 OCR 时，如果在听写开始阶段就提交识别，结束后再按文本长度丢弃结果，虽然输出语义正确，但本地模型加载、远程请求和推理资源已经消耗，长度门槛没有达到节流目的。

正确做法是把窗口上下文拆成两个阶段：

1. 听写开始时完成应用身份、黑名单和密码控件检查，并冻结当时的窗口截图；这样上下文仍对应用户开始听写时看到的内容。
2. 识别文本结束后，使用软件规则解析出的实际 `smartProcessingMinChars` 判断是否执行智能处理。只有 `trim()` 后的 Unicode 字符数达到门槛时才提交 OCR；短文本直接释放截图。

延迟提交 OCR 时必须从实际提交时刻重新创建完整截止时间，不能复用听写开始时的旧截止，也不能沿用只用于读取已完成结果的短等待时间。取消、失败或新会话替换后，OCR、智能处理和文字注入前都要重新校验会话，旧会话结果不得继续写入最近上下文或注入文本。

原生文本提取不受该开关影响；门槛为 `0` 或关闭“跟随处理时机”时保持原有即时捕获行为。调试捕获也保持即时执行，便于独立诊断 OCR。
