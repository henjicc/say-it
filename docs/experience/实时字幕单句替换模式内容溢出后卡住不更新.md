# 实时字幕「单句替换」模式：一句话说长了之后悬浮窗看起来卡住不更新

## 现象

设置为「单句替换」（`subtitle.displayMode === "replace"`，固定只显示 1 行）后，
一句话说得比较长、文字宽度超过一行时，悬浮窗看起来从某个时刻起就不再更新了，
后面说的内容像是没有显示出来。

## 排查过程中的误判

一度怀疑是 `requestAnimationFrame` 被节流导致的（指示器窗口创建时 `focused(false)`，
WebView2 可能把非焦点窗口的 rAF 降频甚至暂停）。为此把 `IndicatorApp.tsx` 里
`renderText` 从"rAF 排队绘制"改成"收到事件就同步绘制"（commit `8f7df25`），
但很快被自己 revert 掉（commit `44a01e3`）——因为这并不是根因，字幕文字状态其实
一直在正常更新，只是没画出来对用户不成立。

## 真正的根因

`applyScroll()`（`ui/src/indicator/IndicatorApp.tsx`）里，`replace` 模式被特殊处理成
**永远强制 `translate3d(0, 0, 0)`，不做任何滚动**：

```ts
if (mode === "subtitle" && subtitleConfig.displayMode === "replace") {
  const next = "translate3d(0, 0, 0)";
  ...
  return; // 不执行下面的溢出滚动逻辑
}
```

而 `syncSubtitleIndicator()`（`ui/src/features/subtitles/controller.ts`）里，
`replace` 模式的框高固定按 `effectiveLines = 1` 计算，不受"显示行数"设置影响。

问题就出在这两者叠加：`#text-content` 用 `white-space: pre-wrap` 换行，一旦当前这句话
渲染宽度超过一行，多出来的文字会自然换到第二行、第三行，但 `#text-flow` /`#text`
都是 `overflow: hidden` 且只留 1 行高度。由于 `applyScroll` 在 replace 模式下把
transform 锁死在 `(0,0,0)`，视图永远只显示"第一行"，超出的后续内容被裁在看不见的
下方——**DOM 里的文字其实一直在正确更新**，只是可见区域没有跟着滚动上去，看起来就像
"填满方框后不再更新"。

## 修复

去掉 `applyScroll` 里 replace 模式的特殊分支，统一用原有的"按溢出量向上滚动"逻辑
（`overflow = scrollHeight - flow.clientHeight`，`offset = max(overflow, 0)`）。
内容没溢出时 offset 自然是 0，效果和之前一致；一旦当前这句话换行溢出单行框，
会像 `scroll` 模式一样把视图滚到能看到最新文字的位置，不会再卡住。

## 教训 / 可复用经验

- 遇到"界面看起来卡住不更新"类问题，不要默认是"状态没更新"或"渲染时机被节流"，
  先确认 DOM/state 是否真的在变化（可以先加日志或临时去掉 CSS `overflow: hidden`
  验证），再决定是状态问题还是纯视觉裁剪/滚动问题。
- 涉及固定高度 + `overflow: hidden` 的展示框，只要内容可能超出固定行数，就必须有
  滚动或截断策略；"永远锁定在顶部不滚动"这类特殊分支要谨慎，容易在内容变长时
  制造出"看起来没更新"的假象。

## 后续调整：改上下滚动为左右滚动（体验优化，非 bug）

上面的修复让 replace 模式"能动"了，但效果是整句换行后**向上滚动**——用户反馈：
说话较快时，上一行还没读完就被新内容顶没了，阅读体验不如"从左到右连续流动"。

`applyScroll()` 现在按 `isReplaceMode` 分两支：

- `replace` 模式：`#text-content` 改 `white-space: nowrap`（CSS 里 `.subtitle-replace
  #text-content { right: auto; white-space: nowrap; }`，去掉右边界让它靠 `left:0`
  收缩到内容自身宽度，即 shrink-to-fit），强制整句单行不换行。JS 按
  `content.scrollWidth` 与 `flow.clientWidth` 的差值算偏移：句子没超宽时居中
  （`offset = (flowWidth - contentWidth) / 2`），超宽后贴右边显示最新内容
  （`offset = -(contentWidth - flowWidth)`），旧内容随句子变长持续向左流出，
  不会出现"一整行被顶掉"的跳变。
- `scroll`（滚动累积）模式：逻辑不变，仍是原来的按 `scrollHeight` 溢出量向上滚动。

**关键点**：单行不换行 + 绝对定位只设 `left`（不设 `width`/`right`）就能让容器
按内容自身宽度收缩（shrink-to-fit），这样才能用 `scrollWidth` 精确测量"这句话
渲染出来到底多宽"，再纯靠 `transform: translate3d(x, 0, 0)` 做视觉平移，不触发
布局重排。这个套路（shrink-to-fit 测宽 + transform 平移）比"设固定宽度再裁剪"
更适合"内容不定长、要连续滚动"的场景，可复用到其它单行走马灯类需求上。

## 再后续：第一个词会从左边"飞"进来

改成左右平移后发现一个新问题：每次从空白状态说出第一个词，它会明显从左边
滑入到居中位置，而不是直接出现在正中间。

原因是 `#text-content` 上有 `transition: transform 0.12s ease-out`（用于日常
逐词更新时的平滑过渡），而清空状态（`resetText()`）把 transform 残留在
`translate3d(0, 0, 0)`（贴左）。第一个词到来时 `applyScroll` 算出的目标是居中
偏移，这个从"贴左"到"居中"的跳变会被同一个 transition 捕获，于是就"飞"了一下。

修复：`renderText` 在 rAF 回调里判断 `isFirstPaint =
!displayedText.current && !!pendingText.current`（即上一次是空白、这一次刚出现
第一批文字），把这个标记传给 `applyScroll(skipAnimation)`。为真时临时把
`content.style.transition` 设为 `"none"`、直接落位、强制 reflow（`void
content.offsetWidth`）后再恢复原 transition，让"从空白到第一次出现"这一帧直接
定位，不经过动画；后续逐词更新仍然走正常的过渡动画。

**可复用经验**：只要一个元素的 transform/位置带了 CSS transition，又存在"从
一个初始残留值跳到刚计算出来的目标值"这种场景（比如首次出现、从隐藏切换到
可见），就要专门判断"是不是这次的第一帧"，对第一帧关掉 transition 直接落位，
否则残留的初始值会被过渡动画放大成一次视觉上莫名其妙的位移。
