# 实时字幕「单句替换」模式：短暂停顿不应该把当前行清空重开

## 需求背景

「单句替换」模式下，识别接口按 VAD（`max_sentence_silence`，设置里叫"断句静音阈值"，
默认约 1300ms）检测到停顿就会把当前话语标记为 `final`，提交进 `committedLines`，
`currentSegment` 清空，下一次识别结果会开启一段全新的话语。

一度以为这是"断句静音阈值"没调好，但用户明确指出这不是识别参数问题：即使把这个
阈值调大，只要说话中间的停顿超过它，识别层面依然会finalize、开启新的一段。真正的
诉求是**软件层面**的展示策略——只要当前这一行内容还不长（没说满一屏），说话中间
随便停顿多久，后面继续说的内容都应该"接着贴在同一行后面"，而不是被新话语整段替换
掉、看起来像清空重来；只有停顿长到可以认为是"换了个话题"（比如 5 秒以上）才应该
重新起一行。

## 实现

`ui/src/features/subtitles/controller.ts` 新增：

- `replaceModeLine` / `replaceModeLineAt`：单句替换模式下"已经说完、不会再变"的
  这一行文本，以及最近一次更新它的时间戳。可能是跨了好几次 VAD 断句拼接起来的。
- `REPLACE_LINE_CONTINUE_GAP_MS = 5000`：新一句话开始时，如果距离上一句 finalize
  的时间没超过这个值，就把 `replaceModeLine` 当作前缀接着用；超过了才清空重开。

关键位置：`handleSubtitleAsrEvent` 里，`result` 事件的第一个 partial（即
`!currentSegment` 为真、说明上一句刚 finalize 完）到达时才做一次"续接 or 清空"的
判断；同一句话内部持续增长的 partial 不重复判断。`final` 时把这句话追加进
`replaceModeLine`（用空格分隔，并做 1800 字上限截断防止无限增长）。

`renderSubtitle` 展示的文本相应改成：有 `nextSegment`（说话中）就是
`replaceModeLine + 分隔符 + nextSegment`；没有（刚 finalize 完）就是
`replaceModeLine` 本身。

## 为什么没做"按显示区域是否填满来判断"

用户最初想法是"按当前行文字是否填满显示区域（比如 < 70%）来决定要不要重开"，
这需要测量真实渲染宽度——但渲染发生在悬浮窗（`ui/src/indicator`）这个独立窗口/
进程里，`controller.ts` 侧根本拿不到 DOM 尺寸。用户自己也提出了"设一个停顿时长
阈值（比如 5 秒）"作为备选方案，实现和推理都远比跨进程测量宽度简单，所以采用了
纯时间阈值的方案。

## 教训 / 可复用经验

- 遇到"识别结果被打断/清空"类反馈，先分清楚这是**识别层**的断句行为（VAD 参数）
  还是**展示层**要不要沿用旧内容的策略——这是两个独立的问题，参数调得再好也解决
  不了展示层"直接整段替换"的观感。
- "是否续接 vs 重新开始"这类状态判断，只应该在状态发生**跳变的那一刻**做一次
  （这里是"上一句刚 finalize、这一句刚开始"），不要在同一状态持续更新的每一帧都
  重复判断，否则容易把判断逻辑和值的来源搞混。
