# 持久化密钥字段必须复用统一掩码与显隐实现

## 触发条件

页面需要展示、覆盖或按需读取已经持久化的 API Key、密码或其他敏感字段。

## 问题根因

如果页面把固定长度的 `•` 掩码直接写入受控 Input 的 `value`，聚焦和输入时就可能把掩码字符与用户新内容一起传入 `onChange`，最终覆盖真实密钥。页面分别实现显隐按钮、编辑态和异步读取，也会导致保存语义、错误处理和样式持续漂移。

## 正确做法

- 统一复用 `ui/src/components/ui/SecretInput.tsx`，页面只持有真实草稿值。
- 已保存掩码通过展示状态或 placeholder 表达，绝不能成为可编辑 value。
- 明文只在用户点击显隐按钮后按需读取；不自动读取，不写日志，不进入普通供应商快照。
- 失焦、保存成功、切换供应商或组件卸载后清除临时明文并恢复掩码。
- 自动保存和手动保存属于业务策略，由页面或 controller 处理；通用组件不直接持久化。
- 新建实体时尚未持久化的密钥可以使用普通 `type="password"` 输入框。

## 验证重点

- 直接聚焦已保存密钥后输入，新草稿中不包含任何掩码字符。
- 点击显隐可以读取真实值，失焦后立即重新打码。
- 不修改密钥时保存其他配置，不应覆盖原密钥。
- 读取失败保留掩码并显示错误，不泄露明文或错误上下文中的敏感数据。
