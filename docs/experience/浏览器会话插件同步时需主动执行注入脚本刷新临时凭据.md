# 浏览器会话插件同步时需主动执行注入脚本刷新临时凭据

## 触发条件

- `browserSession` 插件不是只依赖稳定登录 Cookie，还依赖网页登录页运行时临时生成的短时凭据、签名 URL 或页面内业务态。
- 这些临时值通过 `initializationScript` 写回自定义 Cookie，再由 `syncSession` 保护保存。

## 正确做法

- 不要只依赖“打开登录窗口时注入一次脚本”。
- 在用户点击 `syncSession` 时，宿主要先对当前登录 WebView 再执行一次 `initializationScript`，然后轮询等待必要 Cookie 出现后再保存会话。
- 插件侧注入脚本也不能做成“只要初始化过就永不再跑”；应提供可重复触发的刷新入口，保证每次同步都能生成新的临时凭据。
- 如果临时凭据通过自定义 Cookie 回写，刷新开始前要先同步清掉旧 Cookie；否则宿主只按“Cookie 名存在”判定成功时，可能会把上一轮的旧值抢先保存下来，形成“页面日志显示刷新成功，但插件拿到的仍是旧凭据”的竞态。
- 对这类承载临时凭据的自定义 Cookie，插件应通过通用的 `browserSession.capturedUrlCookie` 声明 Cookie 名、有效期、目标 URL 和必要查询参数；宿主只按该声明校验值本身、字段完整性和时间戳新鲜度，不能在宿主代码里按插件 ID 写分支。

## 原因

- 这类凭据常常有很短的有效期，或者依赖页面脚本完成初始化后才能生成。
- 仅在窗口创建时注入，容易因为执行时机太早、页面脚本尚未就绪或凭据已过期，导致 `syncSession` 保存到的会话缺少关键标记，最终在实时识别启动时失败。
