# 字幕编辑器播放头「暂停后恢复播放的瞬间先顿一下再跳过去」

## 症状

- 字幕编辑页按空格播放，播放头（时间轴上的竖线指针）在**暂停后再次播放**的一瞬间先冻住一小段，然后突然跳到前面一点的位置，之后才恢复丝滑移动。
- 从 0 开始首次播放没有该现象。
- 多轮修复（波形计算切片让出主线程、吞掉 currentTime 微小回退、时间轴滚动改为瞬时）都只解决了各自的次要问题，该现象仍在。

## 根因

播放头每帧直接镜像 `audio.currentTime`，而 Chromium（WebView2）的媒体时钟在 **暂停后 `play()` 恢复时并不连续**：

1. `play()` 后需要重启音频输出管线（Windows 上为 WASAPI sink），期间约 50~200ms 媒体时钟**停滞不动**；
2. 管线就绪后时钟**带补偿地跳步**到实际播放位置。

因此任何「直接读 currentTime 显示」的写法都必然表现为先冻住再前跳，与读取时机无关，无法在读取侧修复。从 0 首播无此现象是因为加载/seek 后管线已预热，时钟从 0 干净起步。

## 修复方式（专业编辑器的标准做法）

播放期间**用 rAF 壁钟推进播放头**，`audio.currentTime` 只作为漂移校准源：

- 每帧 `displayMs += 帧间隔 × playbackRate`，按空格瞬间即匀速起步；
- 与媒体时钟的漂移每帧限幅缓慢追平（速度偏差 ≤15%，视觉不可察觉），同时天然吸收 currentTime 偶发的几毫秒微小回退；
- 漂移超过 300ms 视为真实跳变（主动 seek 等）才硬对齐；
- `seek()` / 暂停 / 换媒体时置空壁钟锚点，下一帧从新的 currentTime 重新起步；
- 暂停/结束事件里仍直接取精确 currentTime，保证拆分、对齐等编辑操作用的是准确位置。

实现见 `ui/src/components/transcription/SubtitleEditor.tsx` 中 `syncPlayhead` / `playheadClockRef`。

## 经验

- 播放头、进度指示等连续动画**不要直接镜像媒体元素时钟**；媒体时钟在 play/pause/seek 边界是分段、跳步的，应以壁钟驱动 + 漂移缓慢校准。
- 同一症状（指针卡顿/跳变）可能有多个独立成因（主线程阻塞、smooth 滚动被高频打断、时钟本身跳步），逐个修掉次要成因后若现象仍在，要回头怀疑数据源本身。
