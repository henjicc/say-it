## 沟通与执行

- DO NOT send optional commentary
- 用制作桌面端软件的思维进行开发
- 默认使用中文回复；用户明确要求其他语言时例外
- 回复聚焦于结论、关键决策、风险和验证结果，避免无关展开
- 除非用户明确要求，否则不要额外创建说明文档、总结文档或示例项目
- 对会显著影响架构、数据、安全或兼容性的歧义，先确认；其余情况做合理假设并明确说明后继续
- 不把本可自行通过代码、日志、测试或文档确认的问题反复抛给用户
- 在遇到问题并解决之后，自动判断是否有必要记录，如果有必要，以 markdown 写入 docs\experience 目录下，不存在则创建该目录，除非找到十分相关的文件需要修改或更新，否则每次都单独创建一个文件，文件名要精准明确概括问题，方便后续查询，后续遇到问题也积极查询该目录是否有相关经验可供参考
- 涉及鼠标操作（拖拽、点击、悬浮、画布交互等需要实际操作 UI 才能验证的场景）的测试，不要自己上手操作和验证；改完后把具体的操作步骤和验证点写清楚，交给用户自己来测试
- 每次改完代码后，需要明确告诉用户这次改动是否需要重启，还是页面会自动热更新；不要默认用户知道，每次都要说明，按照以下格式输出即可，不需要额外解释：✔️无需重启 / ⚠️ 需要重启

## 核心原则

1. **先理解，后修改**
   - 修改前阅读相关代码、测试、配置、项目说明和必要的 Git 历史。
   - 优先理解现有设计意图，不根据文件名或局部代码猜测整体行为。

2. **最小影响**
   - 只修改完成当前任务所必需的内容。
   - 不顺手重构无关模块，不改变无关格式，不扩大任务范围。
   - 尊重现有架构、命名、风格和抽象；引入新模式前先确认现有模式不能满足需求。

3. **解决根因**
   - 修复根本原因，而不是隐藏错误、吞掉异常、关闭检查或堆叠临时补丁。
   - 不为了让测试通过而削弱测试；只有需求确实改变时才同步调整测试预期。

4. **简单且可维护**
   - 优先选择直接、清晰、容易验证的实现。
   - 避免过度设计、过早抽象和不必要的兼容层。
   - 优先复用现有组件、工具函数、服务和约定，避免同一能力出现多份实现。
   - 当多种行为仅由参数或类型差异决定时，优先使用配置、数据或策略驱动，避免散落硬编码分支。

## 工作流程

### 1. 开始前检查

- 查看仓库状态和已有未提交改动，确认哪些改动属于用户。
- 找到项目实际使用的安装、开发、测试、检查和构建命令，不自行臆造。
- 确认生成文件、迁移文件、锁文件和外部依赖的修改规则。

### 2. 非简单任务先规划

以下任务先给出或形成可执行计划：

- 涉及三个以上步骤；
- 跨多个模块或层级；
- 涉及架构、数据结构、公共 API、依赖或迁移；
- 问题原因尚不明确；
- 修改失败的代价较高。

计划应包含：目标、影响范围、关键步骤、风险和验证方式。

### 3. 小步实现

- 将任务拆成可独立验证的小步骤，一次只处理一个明确目标。
- 每完成一个阶段就检查差异和当前行为，避免累积大量未经验证的修改。
- 若事实与计划不符、实现开始偏离目标，或同一路径连续失败两次，立即停止并重新分析，不要继续堆补丁。
- 可使用子代理或并行任务时，只分派彼此独立、边界明确的工作；每个子任务只负责一个目标。

### 4. 完成前验证

- 使用与改动最相关、成本最低的检查开始，再按风险扩大验证范围。
- 常见顺序：目标测试 → 静态检查/类型检查 → 相关测试集 → 构建 → 端到端或运行时验证。
- 修复 Bug 时，在可行情况下补充能复现问题的回归测试。
- UI 或交互修改不能只依赖编译通过；条件允许时实际运行并检查关键流程或截图。
- 检查日志、错误处理、边界情况和失败路径，而不仅是正常路径。
- 未获得验证证据前，不得声称“已完成”“已修复”或“没有问题”。
- 若受环境限制无法执行某项验证，明确说明未验证的内容、原因和剩余风险。

### 5. 最终复查

完成任务或创建提交前检查：

- `git diff` 是否只包含预期改动；
- 是否误改、覆盖或删除了用户已有工作；
- 是否引入重复实现、无用代码、调试输出或临时文件；
- 是否新增不必要依赖、硬编码、全局状态或跨层耦合；
- 行为、测试、类型、文档和配置是否保持一致；
- 实现是否可以进一步简化，但不要为了“优雅”扩大改动范围。

## 代码与架构约束

- 每个模块、类和函数应有清晰职责；当职责难以一句话说明或难以独立测试时，优先拆分。
- 依赖方向必须清晰，底层模块不得反向依赖上层界面或业务编排。
- 业务逻辑不得散落在展示层；外部服务、存储和平台能力应通过明确边界封装。
- 不复制已有逻辑；先搜索现有实现，再决定扩展还是新增。
- 不引入裸类型逃逸、全局禁用检查或无说明的忽略标记；确有必要时写明原因和移除条件。
- 导出的公共接口应保持明确、稳定，并有必要的类型或契约说明。
- 错误必须被正确传播、转换或记录；禁止无声失败。
- 注释解释“为什么”和非显而易见的约束，不复述代码本身。
- 生成产物应通过其源文件或生成命令更新，除非项目明确规定可以手工修改。
- 新增依赖前确认现有依赖或标准库无法合理解决，并考虑体积、维护、安全和许可成本。

## 安全与破坏性操作

未经用户明确授权，不得：

- 删除或覆盖用户未提交的修改；
- 执行强制推送、重写历史、硬重置或大范围清理；
- 删除数据、执行不可逆迁移或批量重命名；
- 提交、输出或记录密钥、令牌、密码及其他敏感信息；
- 修改生产环境、线上资源或真实用户数据；
- 擅自推送、发布或部署；本地 `git commit` 按下方 Git 约定执行。

执行具有破坏性或不可逆风险的操作前，先说明影响范围和恢复方式。

## Git 约定

- 默认允许自主执行本地 `git commit`，无需每次向用户确认；用户明确要求暂不提交或禁止提交时例外。
- 提交应积极但有边界：完成一个独立修改、修复一个问题、实现一个可验证功能、完成一个阶段，或判断当前状态值得保留时，应及时提交，不要长期积累大量未提交改动。
- 每个提交只表达一个明确目的，并保持可独立理解、验证和回退；不要把无关修改捆绑在同一提交中。
- 提交前检查 `git status`、`git diff` 和暂存区内容，只暂存本次任务相关文件，不覆盖、回滚或混入用户已有的无关改动。
- 提交前执行与改动相匹配的必要验证。原则上不提交明显损坏、无法运行或尚未完成的状态；确需建立阶段性检查点时，提交信息必须明确标注“检查点”及当前限制。
- 提交信息使用简洁、准确的中文，直接说明本次改动，例如：`修复：解决首次加载白屏问题`、`新增：支持批量导入素材`、`重构：拆分图层查询与修改逻辑`。
- 大型、高风险或跨阶段修改前，在当前状态完整且适合回退时，可以主动创建检查点提交。
- 若工作区存在无法安全区分归属的用户改动，不要贸然提交；先保留现场，并在结果中说明原因。
- 完成任务后再次检查 `git status`，并在最终回复中说明已创建的提交及仍未提交的改动。
- 未经用户明确授权，不执行 `git push`、强制推送、变基共享分支、修改远端历史、发布或部署。

## 经验沉淀

- 用户纠正后，判断它是一次性偏好还是可复用规则。
- 只有稳定、明确、能够防止同类错误再次发生的经验，才写入项目规则或专门的经验文件。
- 新规则应描述触发条件和正确做法，避免只记录某次事件。
- 定期删除过时、重复、互相冲突或模型本就能从代码推断的规则。
- 本文档只保留每次任务都可能适用的高价值约束；长流程、领域知识和低频操作拆到 `docs/rules/` 中按需加载。
- 必须无例外执行的格式化、检查或安全限制，应尽量交给 Hooks、CI 或脚本确定性执行，而不是只依赖文字提醒。

## 上下文管理

- 不相关任务使用新的会话或清理上下文，避免旧问题污染当前判断。
- 压缩或交接上下文时，保留：用户目标、已修改文件、关键决策、已运行命令、验证结果、未解决问题和风险。
- 不在主上下文中堆积大段低频参考资料；需要时再读取对应文件。

## 项目专属信息

- 项目目标：桌面端语音/听写应用「说吧！」，包含主窗口与悬浮指示窗口。
- 技术栈与运行环境：Tauri 2 + Rust 2021 后端，React 19 + TypeScript + Vite 6 + Tailwind CSS 4 前端，Windows 为主要打包目标（NSIS）。
- 包管理器：npm，锁文件为 `package-lock.json`。
- 安装命令：`npm install`。
- 开发命令：前端独立开发用 `npm run ui:dev`；完整桌面应用开发用 `npm run tauri:dev`。
- 格式化/静态检查命令：当前未在 `package.json` 中配置 lint/format/typecheck 脚本；需要按改动范围使用可用工具，例如 `npx tsc --noEmit` 或 `cargo check`。
- 测试命令：当前未配置自动化测试脚本。
- 构建命令：前端构建用 `npm run ui:build`；桌面应用打包用 `npm run tauri:build`。
- 核心目录与依赖方向：`ui/src` 为前端界面与状态；`src-tauri/src` 为 Tauri/Rust 命令、桌面能力、音频与 ASR 服务；前端通过 `ui/src/lib/tauri.ts`、hooks 和 Tauri invoke/event 与后端交互。
- 不可违反的架构边界：业务与平台能力优先放在 `src-tauri/src` 的明确模块中；React 组件避免直接承载复杂业务流程；多入口页面需同时考虑 `ui/index.html` 主窗口和 `ui/indicator.html` 悬浮窗。
- 自动生成文件及其生成命令：`ui/dist/` 由 `npm run ui:build` 生成；`src-tauri/target/` 由 Cargo/Tauri 生成；`src-tauri/gen/schemas/` 为 Tauri 生成 schema，通常不要手工改。
- 必需环境变量或平台限制：开发端口固定为 `5155` 且 `strictPort: true`；Tauri 配置的 `devUrl` 依赖该端口。
- 已知陷阱和非显而易见行为：`vite.config.ts` 以 `ui/` 为 root，并配置 `main` 与 `indicator` 两个打包入口；`tauri.conf.json` 中主窗口初始 `visible: false`、`decorations: false`，窗口显示逻辑在后端控制；`ui/dist/`、`src-tauri/target/` 不应手工维护。
